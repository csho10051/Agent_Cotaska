## 3. 候補技術の比較

### 3.1 選択肢一覧

| # | 構成 | フロントエンド | バックエンド | DB接続 |
|---|------|--------------|-------------|--------|
| A | **Tauri** | HTML/CSS/JS（React/Vue/Svelte） | Rust | rusqlite / sqlx |
| B | **Electron** | HTML/CSS/JS（React/Vue/Svelte） | Node.js | better-sqlite3 |
| C | **Python GUI** | PyQt6 / PySide6 / Flet | Python | SQLAlchemy + sqlite3 |

### 3.2 評価軸と比較

| 評価軸 | Tauri (A) | Electron (B) | Python GUI (C) |
|--------|:---------:|:------------:|:--------------:|
| **4カラムUI実現性** | ◎ Web技術で自由 | ◎ Web技術で自由 | △ PyQt可だが手間 |
| **Markdownエディタ** | ◎ CodeMirror等 | ◎ CodeMirror等 | △ QScintilla等 |
| **ドラッグ&ドロップ** | ◎ HTML5 D&D | ◎ HTML5 D&D | △ Qt D&D |
| **進捗バー/自動保存** | ○ | ○ | ○ |
| **SQLite連携** | ○ rusqlite | ◎ better-sqlite3 | ◎ SQLAlchemy |
| **バンドルサイズ** | ◎ 5〜10MB | × 100MB〜 | ○ 30〜50MB |
| **メモリ使用量** | ◎ 軽量 | × Chromium常駐 | ○ 中程度 |
| **学習コスト** | △ Rust必要 | ○ JS のみ | ○ Python のみ |
| **エコシステム成熟度** | ○ 成長中 | ◎ 非常に豊富 | ○ GUI寄りは少なめ |
| **AI連携（将来）** | ○ | ○ | ◎ Python直接 |
| **マイグレーション** | △ 独自 or 生SQL | ○ knex migrate | ◎ Alembic |
| **インストール難度** | △ Rust+VS Build Tools | ○ Node.jsのみ | ○ Pythonのみ |

### 3.3 セットアップ手順の比較

#### A. Tauri

1. Rust インストール（rustup）
2. Visual Studio Build Tools インストール（C++ ビルドツール）
3. Node.js インストール（フロント開発用）
4. `cargo install create-tauri-app`
5. Tauri プロジェクト作成

→ **セットアップが最も重い**（Rust + Node.js + VS Build Tools）

#### B. Electron

1. Node.js インストール（nvm-windows or 公式インストーラ）
2. `npm init` + `npm install electron`
3. Electron プロジェクト作成

→ **セットアップが最も軽い**（Node.js のみ）

#### C. Python GUI

1. Python インストール（公式インストーラ or pyenv-win）
2. 仮想環境作成 `python -m venv .venv`
3. `pip install PySide6` or `pip install flet`
4. SQLAlchemy + Alembic インストール

→ **中程度のセットアップ**

## 5. 推奨

### 第1推奨：**B. Electron**

理由：

- CoTasker のUI要件（4カラム、Markdown、D&D）は **Web技術が最適** → Tauri/Electron
- Tauri は Rust + VS Build Tools のセットアップが重く、Rust の学習コストが高い
- Electron は **Node.js のみで開始可能**、エコシステムが最も豊富
- better-sqlite3 による SQLite 連携が成熟
- バンドルサイズ（100MB〜）はローカルアプリとして許容範囲
- 実績多数（VS Code, Slack, Notion Desktop 等）

### 第2推奨：**A. Tauri**（パフォーマンス重視の場合）

理由：

- バンドルサイズとメモリ効率は最優秀
- ただしセットアップと学習コストが最も高い
- 将来的にパフォーマンスが問題になった場合の移行先として有効

### 非推奨：**C. Python GUI**

理由：

- 4カラムUI + Markdownエディタ + D&D の実現が困難
- TickTick レベルのUIを Python GUI で再現するのは非現実的
- AI連携の親和性は高いが、UI要件がボトルネック

## 6. 決定事項（選定後に記入）

選定技術：**Electron**

フロントエンド：**React**

バックエンド：Node.js

DB接続方式：sql.js（WASMベース、ネイティブビルド不要）

マイグレーション方式：生 SQL（init_schema.sql）

決定日：2026-03-16

更新日：2026-03-16

決定理由：

- 4カラムUI・Markdownエディタ・D&DなどCoTaskerのUI要件にWeb技術が最適
- Node.jsのみで開始可能でセットアップが最も軽い
- エコシステムが最も豊富（VS Code, Slack等と同じ基盤）

変更履歴：

- better-sqlite3 → sql.js に変更。better-sqlite3はネイティブビルド（Python + VS Build Tools）が必要で、
  プロキシ環境でのprebuildダウンロードも失敗したため、pure JavaScript実装のsql.jsを採用。
  sql.jsはWASMベースでSQLite相当の機能を提供し、ネイティブビルドが不要。
- フロントエンドFWを React に決定。AIエージェントとの協業でコード生成品質が最も安定、
  Electronとの組み合わせ実績が豊富、エコシステムが最大。
