# タスク管理システム（Cotaska） - システム全体設計

## 1. システムの目的

本システムは、**個人または小規模チーム向けのローカル完結型タスク管理システム**であり、次の目的を持つ。

-   日々のタスクを分かりやすく整理する
-   すべて・今日・明日・直近7日・リスト別にタスクを確認できる
-   タスク詳細を素早く編集できる
-   完了、遅延、ゴミ箱などの状態を明確に管理できる
-   将来的に
    AIエージェント（Codex等）とタスク情報を共有し、整理・補助・更新支援を受けられるようにする

参考システム：TickTick（UIの構成・操作性を参考にする）

------------------------------------------------------------------------

## 2. システムの基本方針

### 2.1 ローカルファースト

-   データはローカルPCに保存する
-   社内ネットワーク制約に対応する
-   ネットワーク未接続でも基本機能を利用可能

### 2.2 デスクトップアプリ前提

-   常用業務で使いやすいローカルアプリ
-   UIは軽量で高速表示を重視
-   Electron + React（Vite）で実装

### 2.3 タスクファイルを正本とする

-   タスクの正本データは `data/tasks/T-XXX.md`（個別Markdownファイル）
-   UIはタスクファイルの内容を表示・更新
-   `_index.yaml`は自動生成されるサマリー（AI・人間とも直接読み写し可能）
-   AIエージェントがファイルを直接編集すると`fs.watch`経由でUIがリアルタイム反映
-   YAML frontmatter が破損したタスクファイルは、通常タスクとして保存・更新せず、診断用の破損タスクとして画面に表示する

### 2.4 AI連携可能な構造

-   AIが読みやすい構造のタスクデータ
-   未完了タスク抽出
-   優先度提案
-   ログ要約などに対応
-   AI が不正な YAML を生成した場合も、Cotaska 全体のタスク表示は止めず、原因ファイルを利用者が発見できるようにする

## 4. 利用シナリオ

### 日常利用

1.  サイドバーまたはナビパネルから表示カテゴリを選択
2.  メインペインにタスク一覧表示
3.  クイック追加バーからタスクを素早く登録
4.  タスクをクリックすると詳細ペインに詳細表示
5.  詳細ペインから親タスクまたは関連サブタスクへ移動できる
6.  編集内容は自動保存でタスクファイルに反映
7.  チェック操作で完了状態変更
8.  メインペインの日付ホバー / 詳細ペインの日付クリックで期限を更新
9.  詳細ペインで Markdown の編集 / プレビューを虫眼鏡アイコンで切り替える
10.  詳細ペインの「新しいアプリで開く」アイコンから対象 Markdown ファイルを既定アプリで開く
11.  詳細ペイン本文のリンククリックで URL / ファイル / フォルダを判定し、既定ブラウザ / 既定アプリ / エクスプローラを起動する
12.  メインペインでドラッグして並び順を調整し、セクション移動時は `due_date` / `progress_status` を自動更新

### 整理利用

-   リストなしにタスクを登録
-   リストへ振り分け
-   遅延タスク確認
-   完了タスク管理
-   ゴミ箱管理

### AI連携利用（将来）

-   未完了タスク抽出
-   期限超過タスク分析
-   Markdownサマリ生成
-   優先順位提案

## 5. システム構成

    デスクトップUI
        ├ サイドバー：機能アイコン（リスト/検索）
      ├ ナビパネル：リスト・タグ選択
      ├ メインペイン：タスク一覧（クイック追加バー + 期限日ホバーピッカー付き）
      └ 詳細ペイン：タスク詳細・編集（進捗・タグ・所属リスト・期限編集・親タスクリンク・関連サブタスク・Markdownプレビュー・既定アプリ起動・本文リンク起動）

    アプリケーション層
        ├ taskService.js       — タスクCRUD + メモリキャッシュ管理
        ├ listService.js       — リストCRUD
        ├ indexService.js      — _index.yaml 自動生成
        ├ watcher.js           — ファイル変更監視（chokidar）
        └ バリデーション層

    データ保存層
        ├ data/tasks/       — 個別タスクファイル（T-XXX.md）
        ├ data/tasks/_index.yaml  — 自動生成インデックス（読み取り専用）
        ├ data/archive/     — 削除済みタスク保存
        └ data/lists.yaml   — リストマスタ

## 5.1 起動時の初期表示（CHG-077）

**目的**：
- Cotaska.exe 起動直後からメイン画面表示までの間、黒い空白ウィンドウを表示しない
- 利用者にアプリが起動中であることを明示する

**表示責務**：
- `BrowserWindow` の `backgroundColor` はロード画面に近い明色とし、HTML読み込み前の黒い空白を避ける
- `src/renderer/index.html` は React の JavaScript がロードされる前に静的なロード画面を表示する
- main process は `startup:progress` イベントと `startup:getProgress` IPC で起動段階、進捗率、処理内容を renderer へ渡す
- `src/renderer/App.jsx` は初回 `tasks:getAll` 完了までロード画面を表示し、進捗率と現在処理中の内容を表示する

**進捗表示**：
- main process はアプリ初期化、設定確認、データフォルダ確認、ウィンドウ準備、タスク読み込み、リスト読み込み、通知/監視準備、renderer 読み込みの各段階を通知する
- renderer は初回タスク一覧取得と画面組み立ての段階を追加表示する
- 進捗率は起動段階を利用者へ伝えるための目安とし、ファイル単位の厳密な読み込み割合は表さない

**終了条件**：
- 初回タスク取得と初期 view model 構築が完了したら、通常の4カラム画面へ切り替える
- 初回タスク取得に失敗した場合は、通常画面へ進まず起動失敗メッセージを表示する

## 5.2 サービス層詳細設計

### taskService.js（タスク CRUD + キャッシュ）

**責務**：
- アプリ起動時に全タスクファイルを読み込みメモリキャッシュ構築
- CRUD操作時にキャッシュと実ファイルを同期
- gray-matter で frontmatter と本文を分離・結合
- YAML パースに失敗したタスクファイルを破損タスクとしてキャッシュへ登録し、正常タスクの読み込みを継続

**主要メソッド**：

```javascript
// 初期化
await openTaskService()
// data/tasks/ 配下の全 .md ファイルを読み込み}
// cache に格納、メイン実行中はメモリで管理

// 取得
getAllTasks()          // delete_flag=0 のタスク
getCompletedTasks()    // status=done のタスク
getTrashedTasks()      // delete_flag=1 のタスク
getTaskById(id)        // 個別タスク
getTasksByList(list)   // リスト別
getTasksByParent(id)   // 親タスク配下

// 作成
addTask(task)
// 1. _index.yaml から next_task_id を採番
// 2. frontmatter 生成
// 3. T-XXXX.md ファイルを書き込み
// 4. キャッシュ更新
// 5. indexService に通知（_index.yaml 再生成）

// 更新
updateTask(id, fields)
// 1. 対象ファイルを読み込み
// 2. frontmatter フィールド更新
// 3. ファイル上書き
// 4. キャッシュ更新
// 5. indexService に通知

// 並び替え（一括）
reorderTasks(orderedIds, options)
// 1. 表示対象の sort_order を再採番
// 2. セクション移動時の自動更新（due_date / progress_status）を適用
// 3. 対象タスクを一括保存
// 4. indexService に通知

// 本文編集（frontmatter 以外）
updateTaskContent(id, content)
// 1. 対象ファイルを読み込み
// 2. 本文を置き換え
// 3. ファイル上書き
// 4. updated_at を自動更新
// 5. indexService に通知

// タスク状態変更
completeTask(id)       // status→done, completed_at 設定
reopenTask(id)        // done→todo, completed_at→null
trashTask(id)         // delete_flag→1, deleted_at 設定
restoreTask(id)       // delete_flag→0, deleted_at→null
deleteTask(id)        // ファイルを archive/ に移動

// 複製
duplicateTask(id)
// 1. 対象タスク読み込み
// 2. id 新規採番、title 末尾に「（コピー）」
// 3. 新規ファイル作成
// 4. キャッシュ更新

// キャッシュ再構築（watcher 経由）
rebuildCache()
// ファイル追加・削除検知時に呼び出し
// キャッシュ全体を再読み込み
```

**破損タスクファイルの扱い（CHG-044）**：

- `data/tasks/*.md` の YAML frontmatter パースに失敗した場合、例外で初期化を停止しない
- パース失敗、frontmatter 区切り不正、`id` 欠落、`id` 重複は破損タスクとして扱う
- 破損タスクは `is_invalid: true`、`validation_error`、`validation_error_line`、`validation_error_column`、`task_file_path` を持つ診断用データとして renderer へ渡す
- 破損タスクは通常の更新 API の対象外とし、Cotaska から自動書き戻ししない
- 対象 Markdown が修正され、次回読み込みで通常タスクとして解釈できた場合、破損タスク表示は解消される

### listService.js（リスト CRUD）

**責務**：
- `30_data/lists.yaml` の読み書き
- リスト作成・削除時の対応タスク処理
- js-yaml による YAML パース・生成

**主要メソッド**：

```javascript
// 初期化
await openListService()
// lists.yaml を読み込み

// 取得
getAllLists()         // 全リスト（ソート: created_at ASC）
getListByName(name)   // 名前検索

// 作成
addList(name, color)
// 1. lists.yaml に追記
// 2. ファイル更新
// 3. watcher 検知 → 変更通知

// 更新
updateList(name, newName, color?)
// リスト名または色を更新

// 削除
deleteList(name)
// 1. 該当リストを lists.yaml から削除
// 2. 所属タスク（list フィールド）を null に更新
// 3. indexService に通知

// タグ管理（T-032以降）
getTags()            // tags セクション から取得
addTag(name, color)  // タグ追加
deleteTag(name)      // タグ削除
```

### indexService.js（_index.yaml 自動生成）

**責務**：
- taskService のキャッシュから `_index.yaml` を生成
- 高速読み取り用のサマリーを提供
- 上書き時の最新化を自動実行

**主要メソッド**：

```javascript
// インデックス再生成
rebuildIndex(cache)
// 1. cache から全タスク抽出
// 2. frontmatter フィールドのみ（本文 除く）を YAML に変換
// 3. last_updated = 現在時刻
// 4. next_task_id = cache の最大 ID + 1
// 5. _index.yaml 上書き

// 構造例：
{
  task_file_roots: [".", "projectA", "projectB"],
  tasks: [
    { id: "T-001", title: "...", status: "todo", ... },
    { id: "T-002", title: "...", status: "doing", ... }
  ],
  last_updated: "2026-03-18T15:30:00",
  next_task_id: 3
}
```

### watcher.js（ファイル変更監視）

**責務**：
- `task_file_roots` で定義された複数ルートを監視（chokidar）
- ファイル追加・変更・削除を検知
- taskService キャッシュを更新
- indexService に通知
- 双方向同期で renderer に通知
- 破損タスクファイルが含まれていても、watcher 経由の再構築は正常タスクの表示を継続する

### 5.2 `_index.yaml` と実ファイルパス管理ルール（CHG-016 / CHG-051）

- タスク `.md` の frontmatter には実ファイルパスを保存しない
- `_index.yaml` の各タスク要素にも `task_file_path` は保持しない
- 実ファイルパスは `taskService` がファイル走査時に `_filePath` などの内部メタデータとして保持する
- 外部アプリ起動やリンク解決は renderer がパスを直接扱わず、`taskId` を main process へ渡して解決する
- 監視対象ルートは `_index.yaml` の `task_file_roots` で管理し、未設定時は `.`（`data/tasks/` 直下）を既定値とする

### 5.3 並び順・移動時更新ルール（CHG-021）

- 並び順は `sort_order` を正とし、ドラッグ有効ビューでは `sort_order` を最優先する
- 今日→明日、明日→今日の移動では `due_date` を移動先日付へ同期する
- 進捗セクションは「未着・仕掛」に統合済み（CHG-028）のため、進捗セクション間のドラッグ移動は発生しない
- `status = done` タスクは進捗セクション移動の対象外とする
- 並び順更新と属性更新は同一操作として保存し、失敗時は部分反映を残さない

**監視対象**：
- `data/tasks/*.md` — タスク追加・更新・削除
- `data/lists.yaml` — リスト変更
- `data/archive/*.md` — アーカイブ操作

**処理フロー**：

```javascript
startWatcher(win)
// 監視対象ファイル変更を検知

// 1. ファイル追加 → taskService.rebuildCache()
// 2. ファイル変更 → taskService コマッシュ更新
// 3. ファイル削除 → taskService キャッシュから削除

// 毎回、最後に：
indexService.rebuildIndex(taskService.cache)
win.webContents.send('tasks:changed')
// → renderer が自動的に tasks:getAll を呼び出し
```

### バリデーション層

各サービスの入力値チェック：

```javascript
// taskService 入力検証
validateTask(task)
// ✓ title: 1-200文字
// ✓ priority: high|medium|normal
// ✓ status: todo|doing|blocked|done
// ✓ progress_status: 未着|仕掛|保留|完了
// ✓ due_date: YYYY-MM-DD形式 または null
// ✓ list: 存在するリスト名 または null
// ✓ parent: 存在するタスクID または null

// taskService 読み込み時検証
// ✓ YAML frontmatter をパースできる
// ✓ id が存在する
// ✓ id が重複しない
// NG の場合は通常タスクではなく is_invalid=true の診断用タスクに変換

// listService 入力検証
validateList(list)
// ✓ name: 1-100文字、重複なし
// ✓ color: 有効な色コード
```

### 6.3 メインペイン（タスク一覧）

役割：選択中の条件に基づくタスク一覧表示

構成要素

-   **ヘッダー**：プロジェクト名 / ビュー名、ソート設定、メニュー
-   **クイック追加バー**：タスク名入力で即座に新規タスク作成。CHG-075 以降は、入力中の日付表現を `due_date` に補正し、末尾の `~` 入力で既存リスト候補を表示して `list` を選択できる
-   **タスクリスト**：セクション分け付きの一覧

表示情報

-   ドラッグハンドル（ホバー時表示）
-   完了チェック
-   タスクタイトル
-   日付（期限超過状態は赤字表示）
-   優先度プルダウン
-   日付ホバー時のカレンダーピッカー
-   破損タスクは薄い赤背景、赤線、警告マーク、`要確認` バッジで表示

操作

-   タスク行の右クリック → コンテキストメニューを表示
-   クイック追加バーで `今日`、`明日`、`YYYY-MM-DD`、`M/D` などの日付表現を含めて Enter → 日付を新規タスクに設定
-   クイック追加バーで `~` 入力 → 既存リスト候補を表示し、選択したリストを新規タスクに設定
-   日付にホバー → 日付ピッカーを表示、選択時に即時保存
-   コンテキストメニューからサブタスク追加、リスト変更、複製、ごみ箱移動を実行
-   サブタスクは親タスク配下に階層表示し、展開 / 折りたたみできる
-   タグ設定は T-010 で有効化予定
-   破損タスクは診断用表示のみとし、完了チェック、右クリックメニュー、日付変更、ドラッグ並び替えは無効

削除導線

-   タスク行ホバー時のごみ箱アイコンは表示しない
-   タスク削除は右クリックメニューの「ごみ箱に移動」から実行する

セクション分け

進捗セクション（「すべて」「リスト」選択時）

-   未着・仕掛（CHG-028 で統合済み）
-   保留（CHG-050。保留タスクがない場合は非表示）
-   完了

日付セクション（「今日」「次の7日間」選択時）

-   ⚠️ 遅延（期限 < 今日 かつ 未完了）
-   ☀️ 今日（親タスクの日付が今日の配下サブタスクを含む）
-   📅 明日
-   📆 以降（7日以内）

各セクションはヘッダークリックで折りたたみ / 展開が可能（CHG-034）。
折りたたみ時はセクション名の横にタスク件数を表示する。

設計意図：

-   TickTickの一覧UIを参考にした
-   クイック追加により最小ステップでタスク登録
-   ホバー時のみドラッグハンドルを表示し、通常時はすっきり

### 6.4 詳細ペイン（タスク詳細・編集）

役割：タスク詳細の表示と編集

構成要素（CHG-033）

-   **ヘッダー**：チェックボックス、タスクタイトル
-   **メタ情報**：日付・優先度・進捗ステータス・タグ（折りたたみ可）
-   **本文**：Markdownエディタ / プレビュー切替
-   **親タスクリンク**：親タスクが存在する場合、詳細本文より上に親タスクへ移動するリンクを表示（CHG-047）
-   **関連サブタスク**：子タスクが存在する場合、詳細本文の下に `▼関連サブタスク` 折りたたみセクションを表示（CHG-047）
-   **フッター**：所属リスト変更、メニュー

表示・編集項目

-   タスク名（Markdownエディタ）
-   日付（日付ピッカー、期限超過状態は赤字）
-   優先度（プルダウン選択）
-   進捗ステータス（未着 / 仕掛 / 保留 / 完了）
-   コンテンツ / メモ（Markdownエディタ / プレビュー切替）
-   親タスクリンク（クリックで親タスクを選択）
-   関連サブタスク（折りたたみ可能、クリックでサブタスクを選択）
-   タグ
-   所属リスト（フッターで変更）
-   破損タスク選択時は、エラー種別、行・列、エラー内容、対象ファイルパスを表示

操作

-   各フィールド編集 → 自動保存
-   日付クリック → 日付ピッカー表示、選択時に即時保存
-   虫眼鏡アイコン（プレビュー） → 本文の編集 / プレビュー切替
-   「新しいアプリで開く」アイコン → 対象タスク Markdown を OS 既定アプリで起動
-   既定アプリ起動に失敗した場合はエラー通知を表示
-   破損タスクは編集不可とし、「新しいアプリで開く」から対象 Markdown を修正する
-   親タスクリンクまたは関連サブタスク行のクリックで、対象タスクを選択して詳細ペインを切り替える（CHG-047）

設計意図：

-   TickTickの詳細ビューを参考にした
-   Markdownエディタでリッチな記述が可能
-   タグと進捗の組み合わせで作業状態を把握
-   Markdownプレビューで本文整形結果を即時確認できる
-   必要時に外部エディタ/ビューアで同一 Markdown を確認できる
-   親子関係を持つタスクでは、詳細ペイン内で親子タスクをたどれる

## 10. 設計上の重要判断

### タスクファイルを正本とする

すべてのデータは `data/tasks/T-XXX.md`（個別 Markdown ファイル）を正本とする。
UI はタスクファイルの内容を表示・更新し、`_index.yaml` はキャッシュから自動生成されるサマリーである。

YAML frontmatter が壊れている場合でも、Cotaska はアプリ全体を停止させない。
破損ファイルは `is_invalid=true` の診断用タスクとして一覧に表示し、利用者が対象ファイルを開いて修正できる状態を保つ。

### 4カラムUI（TickTick参考）

サイドバー + ナビパネル + メインペイン + 詳細ペインの構成。
一覧性と編集性を両立しつつ、機能アクセスの効率を高める。

### クイック追加とMarkdownエディタ

TickTickのクイック追加方式を採用し、タスク登録の手間を最小化。
タスク内容はMarkdownで記述可能にし、リッチな記録を可能にする。

### 進捗率による状態管理

ステータス（todo/doing/blocked/done）に加え、進捗率（0〜100%）で
より細かい作業状況の把握を可能にする。

### ローカル完結

社内制約に対応。

### AI連携前提設計

ログ / 状態遷移 / データ構造を明確化。

## CHG-045 タスク階層上限

- タスクは `parent` による親子関係で最大5階層まで表示する
- 1階層目から5階層目までは親配下に再帰的に表示し、展開/折りたたみ対象とする
- サブタスクを親に持つタスクも、5階層以内であれば正式対応対象とする
- 6階層目以上のタスクは階層上限超過として警告扱いにする
- 6階層目以上のタスクは一覧から除外せず、1階層目と同じ形で表示する
- 5階層目のタスクにはサブタスク追加を許可しない
- 親タスク完了時は、5階層以内の子孫タスクも完了連動する
- 子孫タスクの状態変更時は、祖先タスクまで進捗を再計算する

## CHG-047 詳細ペイン親子タスク導線

- 詳細ペインは既存の `parent` frontmatter と taskService のタスク一覧から親子関係を解決する
- 選択中タスクに有効な親タスクがある場合、詳細本文より上に親タスクリンクを表示する
- 選択中タスクに直下サブタスクがある場合、詳細本文の下に `▼関連サブタスク` セクションを表示する
- `▼関連サブタスク` は詳細ペイン内で折りたたみ可能とし、閉状態では `▶関連サブタスク` を表示する
- 親タスクリンクまたは関連サブタスク行のクリックは、対象タスクの選択変更として扱い、タスクファイルのデータ構造は変更しない
- 本仕様に伴う DB/frontmatter/API の必須追加は行わず、既存の全タスク取得結果と `getTasksByParent(id)` 相当の抽出で実現する

## CHG-057 バージョン情報表示

- 現在バージョンの正本は `20_app/package.json` の `version` とする
- main process が現在バージョンを取得し、preload bridge 経由で renderer へ渡す
- renderer は Node.js API や `package.json` に直接アクセスしない
- 表示場所は設定画面の「アプリ情報」セクションを第一候補とする
- 初期対応では自動更新を行わず、最新版がある場合も利用者確認付きの手動ダウンロード案内に留める

## CHG-067 アプリ情報更新確認

- アプリ情報タブで現在バージョンと公開最新バージョンを比較できるようにする
- 最新バージョン取得元は GitHub Releases または `latest.json` 相当を候補とする
- 自動更新、更新ファイルの自動取得、自動インストールは行わない

## CHG-069 確認付き自動更新

- Windows の更新対象は Cotaska-Portable 版のみとする
- 更新元は GitHub Releases の `Cotaska-Portable.zip` とする
- main process は Portable 版更新処理を制御し、renderer は preload bridge の IPC だけを使う
- アプリ情報では、更新確認、ダウンロード、終了後差し替え、再起動更新を利用者操作で段階実行する
- Cotaska-Portable 以外の実行形態や開発実行では、自動更新を未対応として案内する
- 最新版がある場合は利用者確認後に外部ブラウザでダウンロードページを開く
- ネットワーク取得失敗時も通常利用を妨げず、再試行可能な表示にする

## CHG-068 バックアップ画面初期機能

- バックアップと復元タブでは Cotaska.exe と同じディレクトリの `backup` フォルダを既定保存先とし、利用者操作で手動バックアップを作成できるようにする
- 復元は復元元バックアップzipを利用者が選択し、復元前バックアップを作成してからタスク正本、リストマスタ、設定ファイルを戻す
- 対象はタスク正本 `data/tasks/*.md`、`data/lists.yaml`、正式な設定保存ファイルとする
- `_index.yaml` は生成物のためバックアップ対象外とし、必要に応じて再生成する
- バックアップ処理は対象ファイルを読み取ってバックアップ先へzipアーカイブするだけとし、タスク正本へ書き戻さない
- 復元実行前には確認ダイアログを表示し、復元後はタスクキャッシュと `_index.yaml` を再構築する
