# タスク管理システム（Cotaska） - システム全体設計

## 1. システムの目的

本システムは、**個人または小規模チーム向けのローカル完結型タスク管理システム**であり、次の目的を持つ。

-   日々のタスクを分かりやすく整理する
-   すべて・今日・明日・直近7日・リスト別にタスクを確認できる
-   タスク詳細を素早く編集できる
-   完了、遅延、ゴミ箱などの状態を明確に管理できる
-   将来的に
    AIエージェント（Codex等）とタスク情報を共有し、整理・補助・更新支援を受けられるようにする

参考システム：TickTick（UIの構成・操作性を参考にする）

## 2. システムの基本方針

### 2.1 ローカルファースト

-   データはローカルPCに保存する
-   社内ネットワーク制約に対応する
-   ネットワーク未接続でも基本機能を利用可能

### 2.2 デスクトップアプリ前提

-   常用業務で使いやすいローカルアプリ
-   UIは軽量で高速表示を重視
-   Electron + React（Vite）で実装

### 2.3 タスクファイルを正本とする

-   タスクの正本データは `data/tasks/T-XXX.md`（個別Markdownファイル）
-   UIはタスクファイルの内容を表示・更新
-   `_index.yaml`は自動生成されるサマリー（AI・人間とも直接読み写し可能）
-   AIエージェントがファイルを直接編集すると`fs.watch`経由でUIがリアルタイム反映

### 2.4 AI連携可能な構造

-   AIが読みやすい構造のタスクデータ
-   未完了タスク抽出
-   優先度提案
-   ログ要約などに対応

## 4. 利用シナリオ

### 日常利用

1.  サイドバーまたはナビパネルから表示カテゴリを選択
2.  メインペインにタスク一覧表示
3.  クイック追加バーからタスクを素早く登録
4.  タスクをクリックすると詳細ペインに詳細表示
5.  編集内容は自動保存でタスクファイルに反映
6.  チェック操作で完了状態変更
7.  メインペインの日付ホバー / 詳細ペインの日付クリックで期限を更新
8.  詳細ペインで Markdown の編集 / プレビューを虫眼鏡アイコンで切り替える
9.  詳細ペインの「新しいアプリで開く」アイコンから対象 Markdown ファイルを既定アプリで開く
10.  詳細ペイン本文のリンククリックで URL / ファイル / フォルダを判定し、既定ブラウザ / 既定アプリ / エクスプローラを起動する
11.  メインペインでドラッグして並び順を調整し、セクション移動時は `due_date` / `progress_status` を自動更新

### 整理利用

-   リストなしにタスクを登録
-   リストへ振り分け
-   遅延タスク確認
-   完了タスク管理
-   ゴミ箱管理

### AI連携利用（将来）

-   未完了タスク抽出
-   期限超過タスク分析
-   Markdownサマリ生成
-   優先順位提案

## 5. システム構成

    デスクトップUI
        ├ サイドバー：機能アイコン（リスト/検索）
      ├ ナビパネル：リスト・タグ選択
      ├ メインペイン：タスク一覧（クイック追加バー + 期限日ホバーピッカー付き）
      └ 詳細ペイン：タスク詳細・編集（進捗・タグ・所属リスト・期限編集・Markdownプレビュー・既定アプリ起動・本文リンク起動）

    アプリケーション層
        ├ taskService.js       — タスクCRUD + メモリキャッシュ管理
        ├ listService.js       — リストCRUD
        ├ indexService.js      — _index.yaml 自動生成
        ├ watcher.js           — ファイル変更監視（chokidar）
        └ バリデーション層

    データ保存層
        ├ data/tasks/       — 個別タスクファイル（T-XXX.md）
        ├ data/tasks/_index.yaml  — 自動生成インデックス（読み取り専用）
        ├ data/archive/     — 削除済みタスク保存
        └ data/lists.yaml   — リストマスタ

## 5.1 サービス層詳細設計

### taskService.js（タスク CRUD + キャッシュ）

**責務**：
- アプリ起動時に全タスクファイルを読み込みメモリキャッシュ構築
- CRUD操作時にキャッシュと実ファイルを同期
- gray-matter で frontmatter と本文を分離・結合

**主要メソッド**：

```javascript
// 初期化
await openTaskService()
// data/tasks/ 配下の全 .md ファイルを読み込み}
// cache に格納、メイン実行中はメモリで管理

// 取得
getAllTasks()          // delete_flag=0 のタスク
getCompletedTasks()    // status=done のタスク
getTrashedTasks()      // delete_flag=1 のタスク
getTaskById(id)        // 個別タスク
getTasksByList(list)   // リスト別
getTasksByParent(id)   // 親タスク配下

// 作成
addTask(task)
// 1. _index.yaml から next_task_id を採番
// 2. frontmatter 生成
// 3. T-XXXX.md ファイルを書き込み
// 4. キャッシュ更新
// 5. indexService に通知（_index.yaml 再生成）

// 更新
updateTask(id, fields)
// 1. 対象ファイルを読み込み
// 2. frontmatter フィールド更新
// 3. ファイル上書き
// 4. キャッシュ更新
// 5. indexService に通知

// 並び替え（一括）
reorderTasks(orderedIds, options)
// 1. 表示対象の sort_order を再採番
// 2. セクション移動時の自動更新（due_date / progress_status）を適用
// 3. 対象タスクを一括保存
// 4. indexService に通知

// 本文編集（frontmatter 以外）
updateTaskContent(id, content)
// 1. 対象ファイルを読み込み
// 2. 本文を置き換え
// 3. ファイル上書き
// 4. updated_at を自動更新
// 5. indexService に通知

// タスク状態変更
completeTask(id)       // status→done, completed_at 設定
reopenTask(id)        // done→todo, completed_at→null
trashTask(id)         // delete_flag→1, deleted_at 設定
restoreTask(id)       // delete_flag→0, deleted_at→null
deleteTask(id)        // ファイルを archive/ に移動

// 複製
duplicateTask(id)
// 1. 対象タスク読み込み
// 2. id 新規採番、title 末尾に「（コピー）」
// 3. 新規ファイル作成
// 4. キャッシュ更新

// キャッシュ再構築（watcher 経由）
rebuildCache()
// ファイル追加・削除検知時に呼び出し
// キャッシュ全体を再読み込み
```

### listService.js（リスト CRUD）

**責務**：
- `30_data/lists.yaml` の読み書き
- リスト作成・削除時の対応タスク処理
- js-yaml による YAML パース・生成

**主要メソッド**：

```javascript
// 初期化
await openListService()
// lists.yaml を読み込み

// 取得
getAllLists()         // 全リスト（ソート: created_at ASC）
getListByName(name)   // 名前検索

// 作成
addList(name, color)
// 1. lists.yaml に追記
// 2. ファイル更新
// 3. watcher 検知 → 変更通知

// 更新
updateList(name, newName, color?)
// リスト名または色を更新

// 削除
deleteList(name)
// 1. 該当リストを lists.yaml から削除
// 2. 所属タスク（list フィールド）を null に更新
// 3. indexService に通知

// タグ管理（T-032以降）
getTags()            // tags セクション から取得
addTag(name, color)  // タグ追加
deleteTag(name)      // タグ削除
```

### indexService.js（_index.yaml 自動生成）

**責務**：
- taskService のキャッシュから `_index.yaml` を生成
- 高速読み取り用のサマリーを提供
- 上書き時の最新化を自動実行

**主要メソッド**：

```javascript
// インデックス再生成
rebuildIndex(cache)
// 1. cache から全タスク抽出
// 2. frontmatter フィールドのみ（本文 除く）を YAML に変換
// 3. last_updated = 現在時刻
// 4. next_task_id = cache の最大 ID + 1
// 5. _index.yaml 上書き

// 構造例：
{
  task_file_roots: [".", "projectA", "projectB"],
  tasks: [
    { id: "T-001", title: "...", status: "todo", task_file_path: "T-001.md", ... },
    { id: "T-002", title: "...", status: "doing", task_file_path: "projectA/T-002.md", ... }
  ],
  last_updated: "2026-03-18T15:30:00",
  next_task_id: 3
}
```

### watcher.js（ファイル変更監視）

**責務**：
- `task_file_roots` で定義された複数ルートを監視（chokidar）
- ファイル追加・変更・削除を検知
- taskService キャッシュを更新
- indexService に通知
- 双方向同期で renderer に通知

### 5.2 `_index.yaml` のパス管理ルール（CHG-016）

- 各タスクに `task_file_path` を保持する（`data/tasks/` からの相対パス）
- パス区切りは `/` で正規化する
- `task_file_path` 未設定の既存タスクは起動時に補完して再保存する
- 監視対象ルートは `_index.yaml` の `task_file_roots` で管理し、未設定時は `.`（`data/tasks/` 直下）を既定値とする

### 5.3 並び順・移動時更新ルール（CHG-021）

- 並び順は `sort_order` を正とし、ドラッグ有効ビューでは `sort_order` を最優先する
- 今日→明日、明日→今日の移動では `due_date` を移動先日付へ同期する
- 進捗セクションは「未着・仕掛」に統合済み（CHG-028）のため、進捗セクション間のドラッグ移動は発生しない
- `status = done` タスクは進捗セクション移動の対象外とする
- 並び順更新と属性更新は同一操作として保存し、失敗時は部分反映を残さない

**監視対象**：
- `data/tasks/*.md` — タスク追加・更新・削除
- `data/lists.yaml` — リスト変更
- `data/archive/*.md` — アーカイブ操作

**処理フロー**：

```javascript
startWatcher(win)
// 監視対象ファイル変更を検知

// 1. ファイル追加 → taskService.rebuildCache()
// 2. ファイル変更 → taskService コマッシュ更新
// 3. ファイル削除 → taskService キャッシュから削除

// 毎回、最後に：
indexService.rebuildIndex(taskService.cache)
win.webContents.send('tasks:changed')
// → renderer が自動的に tasks:getAll を呼び出し
```

### バリデーション層

各サービスの入力値チェック：

```javascript
// taskService 入力検証
validateTask(task)
// ✓ title: 1-200文字
// ✓ priority: high|medium|normal
// ✓ status: todo|doing|blocked|done
// ✓ progress: 0-100 整数
// ✓ due_date: YYYY-MM-DD形式 または null
// ✓ list: 存在するリスト名 または null
// ✓ parent: 存在するタスクID または null

// listService 入力検証
validateList(list)
// ✓ name: 1-100文字、重複なし
// ✓ color: 有効な色コード
```

### 6.3 メインペイン（タスク一覧）

役割：選択中の条件に基づくタスク一覧表示

構成要素

-   **ヘッダー**：プロジェクト名 / ビュー名、ソート設定、メニュー
-   **クイック追加バー**：タスク名入力で即座に新規タスク作成
-   **タスクリスト**：セクション分け付きの一覧

表示情報

-   ドラッグハンドル（ホバー時表示）
-   完了チェック
-   タスクタイトル
-   期限（期限超過は赤字表示）
-   優先度プルダウン
-   期限ホバー時のカレンダーピッカー

操作

-   タスク行の右クリック → コンテキストメニューを表示
-   期限にホバー → 日付ピッカーを表示、選択時に即時保存
-   コンテキストメニューからサブタスク追加、リスト変更、複製、ごみ箱移動を実行
-   サブタスクは親タスク配下に階層表示し、展開 / 折りたたみできる
-   タグ設定は T-010 で有効化予定

削除導線

-   タスク行ホバー時のごみ箱アイコンは表示しない
-   タスク削除は右クリックメニューの「ごみ箱に移動」から実行する

セクション分け

進捗セクション（「すべて」「リスト」選択時）

-   未着・仕掛（CHG-028 で統合済み）
-   完了

日付セクション（「今日」「次の7日間」選択時）

-   ⚠️ 遅延（期限 < 今日 かつ 未完了）
-   ☀️ 今日（親タスク期限が今日の配下サブタスクを含む）
-   📅 明日
-   📆 以降（7日以内）

各セクションはヘッダークリックで折りたたみ / 展開が可能（CHG-034）。
折りたたみ時はセクション名の横にタスク件数を表示する。

設計意図：

-   TickTickの一覧UIを参考にした
-   クイック追加により最小ステップでタスク登録
-   ホバー時のみドラッグハンドルを表示し、通常時はすっきり

### 6.4 詳細ペイン（タスク詳細・編集）

役割：タスク詳細の表示と編集

構成要素（CHG-033）

-   **ヘッダー**：チェックボックス、タスクタイトル
-   **メタ情報**：期限・優先度・進捗ステータス・タグ（折りたたみ可）
-   **本文**：Markdownエディタ / プレビュー切替
-   **フッター**：所属リスト変更、メニュー

表示・編集項目

-   タスク名（Markdownエディタ）
-   期限（日付ピッカー、期限超過は赤字）
-   優先度（プルダウン選択）
-   進捗ステータス（未着 / 仕掛 / 完了）
-   コンテンツ / メモ（Markdownエディタ / プレビュー切替）
-   サブタスク（将来）
-   タグ
-   所属リスト（フッターで変更）

操作

-   各フィールド編集 → 自動保存
-   期限日クリック → 日付ピッカー表示、選択時に即時保存
-   虫眼鏡アイコン（プレビュー） → 本文の編集 / プレビュー切替
-   「新しいアプリで開く」アイコン → 対象タスク Markdown を OS 既定アプリで起動
-   既定アプリ起動に失敗した場合はエラー通知を表示

設計意図：

-   TickTickの詳細ビューを参考にした
-   Markdownエディタでリッチな記述が可能
-   タグと進捗の組み合わせで作業状態を把握
-   Markdownプレビューで本文整形結果を即時確認できる
-   必要時に外部エディタ/ビューアで同一 Markdown を確認できる

## 10. 設計上の重要判断

### タスクファイルを正本とする

すべてのデータは `data/tasks/T-XXX.md`（個別 Markdown ファイル）を正本とする。
UI はタスクファイルの内容を表示・更新し、`_index.yaml` はキャッシュから自動生成されるサマリーである。

### 4カラムUI（TickTick参考）

サイドバー + ナビパネル + メインペイン + 詳細ペインの構成。
一覧性と編集性を両立しつつ、機能アクセスの効率を高める。

### クイック追加とMarkdownエディタ

TickTickのクイック追加方式を採用し、タスク登録の手間を最小化。
タスク内容はMarkdownで記述可能にし、リッチな記録を可能にする。
