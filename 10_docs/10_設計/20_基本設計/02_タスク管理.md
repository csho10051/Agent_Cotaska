# 基本設計 - タスク管理

## 1. 概要

タスクの作成・更新・完了・削除・復元を管理する、システムの中核機能。
クイック追加バーによる素早い登録と、Markdownコンテンツによるリッチな編集を提供する。

**アーキテクチャ（2026-03-18 改訂）**：
- 各タスクは個別の **Markdownファイル**（`30_data/tasks/T-XXX.md`）として保存される
- YAMLフロントマターでメタデータ（id, title, status等）を管理
- Markdown本文がタスクのコンテンツエリア
- `_index.yaml` は自動生成されるサマリーで、読み取り専用
- `_index.yaml` の各タスク要素に `task_file_path` を保持し、分散配置時の参照先を特定する

## 2. タスクファイルのフォーマット

### 2.1 形式

```markdown
---
id: T-001
title: タスク名
status: todo|doing|blocked|done
priority: high|medium|normal
progress_status: 未着|仕掛|完了

progress: 0
due_date: 2026-03-20
list: リスト名
parent: null
tags: []
sort_order: 10
delete_flag: 0
created_at: 2026-03-18T10:00:00
updated_at: 2026-03-18T10:00:00
completed_at: null
deleted_at: null
---
タスク本文（Markdown自由記述）
```

### 2.2 フロントマターフィールド一覧

| フィールド | 型 | 必須 | 説明 |
|-----------|------|:---:|------|
| id | string | ○ | タスクID（自動採番）例：T-001 |
| title | string | ○ | タスク名 |
| status | string | ○ | 状態（todo / doing / blocked / done） |
| priority | string | ○ | 優先度（high / medium / normal） |
| progress_status | string | ○ | 進捗ステータス（未着 / 仕掛 / 完了） |
| progress | number | ○ | 進捗率（0〜100） |
| due_date | string\|null | - | 期限日（YYYY-MM-DD） |
| list | string\|null | - | 所属リスト名（null = リストなし） |
| parent | string\|null | - | 親タスクID。null:親タスク、値がある場合:サブタスク |
| tags | array | ○ | タグリスト（例：["urgent", "client"]） |
| task_file_path | string | ○ | `30_data/tasks/` からの相対パス（例：`projectA/T-001.md`） |
| sort_order | number | ○ | 一覧内の表示順 |
| delete_flag | number | ○ | ゴミ箱フラグ（0:通常, 1:ゴミ箱） |
| created_at | string | ○ | 作成日時（ISO 8601形式） |
| updated_at | string | ○ | 更新日時（ISO 8601形式） |
| completed_at | string\|null | - | 完了日時（ISO 8601形式） |
| deleted_at | string\|null | - | ゴミ箱移動日時（ISO 8601形式） |

### 2.3 本文エリア

フロントマター以下のMarkdownコンテンツを記述。リッチな記述が可能：

```markdown
---
...frontmatter...
---

## 詳細説明

タスクの背景や実装ガイドをMarkdownで記述

- チェックリスト項目1
- チェックリスト項目2

### コード例

```javascript
// 実装例
```

### 2.4 パス解決ルール（CHG-016）

- `_index.yaml.tasks[].task_file_path` を優先して実ファイルの参照先を決定する
- `task_file_path` が未設定の場合は `id.md` をフォールバック参照する
- 起動時またはキャッシュ再構築時に `task_file_path` 未設定タスクを補完し、インデックスへ反映する

### 参考
- [リンク](https://example.com)
```

## 3. ステータス定義

| ステータス | 意味 | 用途 |
|-----------|------|------|
| todo | 未着手 | 新規作成時のデフォルト |
| doing | 作業中 | ユーザーが明示的に変更 |
| blocked | ブロック中 | 外部依存により進行不可 |
| done | 完了 | チェックボックスで自動遷移 |

### 状態遷移図

```text
[新規作成] → todo
               │
               ├──→ doing ←──→ blocked
               │      │
               └──────┴──→ done（完了）
                         ↑完了チェック
                         └再度开始で todo に戻す
```

### 遷移ルール

- どのステータスからでも他のステータスへ自由に遷移可能
- チェックボックス操作（✓）：`status → done` / `done → todo`
- ステータス自体は詳細ペイン下部のドロップダウンで変更する

## 4. 機能一覧

### 4.1 タスク作成（クイック追加）

| 項目 | 内容 |
|------|------|
| トリガー | メインペイン上部のクイック追加バーにタスク名を入力しEnter |
| 入力 | タスク名（必須） |
| 初期値 | status: todo, priority: medium, delete_flag: 0, progress_status: 未着 |
| ビュー連動初期値 | 「今日」では due_date=今日、「明日」では due_date=明日、「次の7日間」では due_date=今日 を初期設定し、「すべて」では due_date=null |
| 保存先 | `30_data/tasks/T-XXX.md`（新規ファイル） |
| 処理 | taskService.addTask() で ID 採番、frontmatter 生成、ファイル書き込み |
| 自動実行 | indexService がキャッシュから `_index.yaml` を再生成 |
| UI反映 | watcher が変更を検知 → renderer に通知 → リスト更新 |
| 設計意図 | TickTickのクイック追加方式を参考。最小ステップでタスク登録 |

### 4.2 タスク更新

| 項目 | 内容 |
|------|------|
| トリガー | 詳細ペインで項目を編集（フォーカスアウト） |
| 対象フィールド | title, due_date, priority, list, progress_status, tags |
| 本文編集 | Markdownコンテンツエリアを直接編集 |
| 期限変更 UI | メインペインでは期限表示ホバー、詳細ペインでは期限表示クリックで日付ピッカーを開き、選択時に即時保存 |
| Markdownプレビュー | 詳細ペイン本文は編集モード / プレビューモードを切り替え可能 |
| タスクID表示 | 詳細ペインフッター右端に `ID: T-XXXX` 形式で常時表示する（CHG-031） |
| 保存先 | 対象タスクファイル（frontmatter + 本文を上書き） |
| 処理 | taskService.updateTask() で frontmatter を更新して上書き |
| 自動実行 | indexService が `_index.yaml` を再生成 |
| UI反映 | watcher が通知 → リアルタイム反映 |

### 4.3 タスク完了

| 項目 | 内容 |
|------|------|
| トリガー | チェックボックスクリック（✓） |
| 処理 | status: done, completed_at: 現在日時を frontmatter に書き込み |
| 処理 | taskService が watcher 経由で自動実行 |
| カスケード完了 | 親タスク完了時、未完了サブタスクも一括で status: done に更新（CHG-009） |
| 親進捗自動推定 | 直下サブタスクがすべて完了なら親も progress_status: 完了 / status: done に更新（CHG-012） |
| 表示 | 一覧から除去（完了一覧に移動）、行が打ち消し線・薄色表示 |
| UI反映 | watcher → リアルタイム反映 |

### 4.4 完了取消

| 項目 | 内容 |
|------|------|
| トリガー | 完了一覧でチェックを外す |
| 処理 | status: todo, completed_at: null を frontmatter に書き込み |
| 処理 | progress_status が「完了」なら「仕掛」に自動変更 |
| カスケード取消 | 親タスクの progress_status を「完了」から戻した場合、直下サブタスクの progress_status も同じ方向に戻す（CHG-010） |
| 親進捗自動推定 | 直下サブタスクに「仕掛」または「完了」が1件でも残る場合、親は progress_status: 仕掛 に再推定される（CHG-012） |
| 表示 | 一覧の該当ビューに戻す |
| UI反映 | watcher → リアルタイム反映 |

### 4.5 ゴミ箱移動

| 項目 | 内容 |
|------|------|
| トリガー | タスク行右クリック「ごみ箱に移動」 |
| 補足 | タスク行ホバー時のごみ箱アイコンは表示しない |
| 処理 | delete_flag: 1, deleted_at: 現在日時を frontmatter に書き込み |
| 処理 | taskService.trashTask() で実行 |
| 表示 | 一覧から除去（ゴミ箱に移動） |
| UI反映 | watcher → リアルタイム反映 |

### 4.6 復元

| 項目 | 内容 |
|------|------|
| トリガー | ゴミ箱一覧で「復元」操作 |
| 処理 | delete_flag: 0, deleted_at: null を frontmatter に書き込み |
| 処理 | taskService.restoreTask() で実行 |
| 表示 | 元のビューに復帰 |
| UI反映 | watcher → リアルタイム反映 |

### 4.7 完全削除

| 項目 | 内容 |
|------|------|
| トリガー | ゴミ箱一覧で「完全削除」操作 |
| 処理 | ファイルを `30_data/archive/` に移動（削除ではなく保存） |
| 処理 | taskService.deleteTask() で実行 |
| 確認 | 確認ダイアログ表示 |
| UI反映 | watcher → リアルタイム反映 |

### 4.8 タスク右クリックメニュー

タスク一覧の各タスク行を右クリックすると、コンテキストメニューを表示する。

| 項目 | 内容 |
|------|------|
| 表示条件 | 通常のタスク一覧でのみ表示。検索ビュー・完了一覧・ゴミ箱では無効 |
| 削除導線 | ごみ箱移動は右クリックメニューからのみ実行し、ホバー時ボタンは表示しない |
| サブタスクの追加 | 親タスク直下にインライン入力欄表示、Enter でサブタスク作成 |
| リスト設定 | サブメニューでリスト一覧表示、選択で `list` フィールド変更 |
| タグ設定 | frontmatter `tags: []` を編集（T-032以降） |
| 複製する | タスク全体を複製、タイトル末尾に「（コピー）」を付与 |
| ごみ箱に移動 | delete_flag: 1 に変更 |

### 4.9 サブタスク作成

| 項目 | 内容 |
|------|------|
| データ構造 | サブタスクは独立した `.md` ファイル、frontmatter に `parent: T-XXX` を設定 |
| 入力UI | 親タスク直下にインライン入力欄を表示 |
| 表示 | メインペインで親タスク直下に階層表示 |
| 展開状態 | 親タスクごとに展開・折りたたみを切り替え可能 |
| 親進捗反映 | 子タスクを持つ親は、子の `progress_status` 変化に応じて親の `progress_status` / `status` を自動再計算 |
| 補足 | サブタスク追加時は親タスクが自動展開される |

### 4.10 タスク複製

| 項目 | 内容 |
|------|------|
| トリガー | タスク右クリック「複製する」 |
| コピー対象 | title, due_date, priority, list, parent, tags（本文は除く） |
| 初期化項目 | status: todo, completed_at: null |
| タイトル | 元のタイトル末尾に「（コピー）」を付与 |
| 処理 | taskService.duplicateTask(id) で実行 |

### 4.11 タスクのリスト変更

| 項目 | 内容 |
|------|------|
| トリガー | タスク右クリック「リスト設定」 → リスト選択 |
| 操作 | frontmatter `list` フィールドを選択リスト名に変更 |
| 未割当 | リストなしを選択した場合は `list: null` |
| 処理 | taskService.updateTask(id, { list }) で実行 |

### 4.12 タスクのドラッグ並び替え / セクション移動（CHG-021）

| 項目 | 内容 |
|------|------|
| トリガー | メインペインのタスク行をドラッグ＆ドロップ |
| 対象ビュー | すべて / 今日 / 明日 / 次の7日間 / リスト / リストなし（完了・ゴミ箱・検索は対象外） |
| 並び替え | ドロップ位置に応じて `sort_order` を更新し、同一表示集合で連番再採番 |
| 日付自動更新 | 今日→明日で `due_date=明日`、明日→今日で `due_date=今日` |
| 進捗自動更新 | CHG-028 により「未着・仕掛」は統合セクションのため、セクション間移動による `progress_status` 自動変更は発生しない |
| 逆遷移 | 日付セクション間の逆方向（明日→今日）は同じルールで適用 |
| 整合制約 | `status=done` タスクは未着/仕掛へのドラッグ変更を禁止 |
| 親子制約 | 親子関係の変更は行わず、初期スコープでは同一親配下の並び替えのみ許可 |
| 失敗時 | 並び順更新と属性更新を一体保存し、部分成功を残さない |

## 5. サービス層インターフェース（taskService.js）

### 初期化

```javascript
await taskService.openTaskService()
// 起動時に 30_data/tasks/ の全ファイルを読み込み、メモリキャッシュを構築
```

### タスク取得

```javascript
tasks = taskService.getAllTasks()
// delete_flag = 0 のタスクを返す

completed = taskService.getCompletedTasks()  
// status = done のタスクを返す（ソート: completed_at DESC）

trashed = taskService.getTrashedTasks()
// delete_flag = 1 のタスクを返す
```

### タスク操作

```javascript
newTask = taskService.addTask({
  title: string,
  due_date?: string,  // YYYY-MM-DD
  priority?: string,  // high|medium|normal
  list?: string,
  parent?: string,    // T-XXX
  tags?: string[]
})
// 新規タスク作成、ID自動採番、ファイル + キャッシュ + _index.yaml を更新

taskService.updateTask(id, {
  title?: string,
  status?: string,
  priority?: string,
  progress?: number,
  due_date?: string,
  list?: string,
  progress_status?: string,
  sort_order?: number,
  tags?: string[]
  // 本文は含まない（別途編集）
})
// 指定フィールドのみ更新

taskService.reorderTasks({
  orderedIds: string[],
  context?: {
    view: string,
    fromSection?: string,
    toSection?: string
  },
  applyAutoFields?: {
    due_date?: string|null,
    progress_status?: string
  }
})
// 並び順更新と必要な属性自動更新を一括保存

taskService.trashTask(id)      // delete_flag = 1
taskService.restoreTask(id)    // delete_flag = 0
taskService.deleteTask(id)     // ファイルを archive/ に移動

taskService.duplicateTask(id)  // タスク複製
```

### その他

```javascript
task = taskService.getTaskById(id)
tasks = taskService.getTasksByList(listName)
tasks = taskService.getTasksByParent(parentId)
```

## 7. バリデーション

| 項目 | ルール |
|------|--------|
| title | 1文字以上、200文字以内 |
| priority | high / medium / normal のいずれか |
| progress | 0〜100の整数 |
| status | todo / doing / blocked / done のいずれか |
| due_date | YYYY-MM-DD形式、過去日も許容、null 可 |
| list | 存在するリスト名または null |
| parent | 既存タスクID（T-XXX形式）または null |
| delete_flag | 0 または 1 |
| progress_status | 未着 / 仕掛 / 完了 のいずれか |

- 削除済みタスク（delete_flag=1）を親タスクとして指定してはならない
- サブタスクのサブタスクは非対応（parent が null のみ親タスク）
