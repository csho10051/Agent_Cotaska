# 基本設計 - タスク管理

## 1. 概要

タスクの作成・更新・完了・削除・復元を管理する、システムの中核機能。
クイック追加バーによる素早い登録と、Markdownコンテンツによるリッチな編集を提供する。

**アーキテクチャ（2026-03-18 改訂）**：
- 各タスクは個別の **Markdownファイル**（`30_data/tasks/T-XXX.md`）として保存される
- YAMLフロントマターでメタデータ（id, title, status等）を管理
- Markdown本文がタスクのコンテンツエリア
- `_index.yaml` は自動生成されるサマリーで、読み取り専用
- ファイル格納パスはタスクデータへ保存せず、アプリが読み込み時に実行時メタデータとして解決する（CHG-051）

## 2. タスクファイルのフォーマット

### 2.1 形式

```markdown
---
id: T-001
title: タスク名
status: todo|doing|blocked|done
priority: high|medium|normal
progress_status: 未着|仕掛|保留|完了
due_date: 2026-03-20
list: リスト名
parent: null
tags: []
sort_order: 10
delete_flag: 0
created_at: 2026-03-18T10:00:00
updated_at: 2026-03-18T10:00:00
completed_at: null
deleted_at: null
---
タスク本文（Markdown自由記述）
```

### 2.2 フロントマターフィールド一覧

| フィールド | 型 | 必須 | 説明 |
|-----------|------|:---:|------|
| id | string | ○ | タスクID（自動採番）例：T-001 |
| title | string | ○ | タスク名 |
| status | string | ○ | 状態（todo / doing / blocked / done） |
| priority | string | ○ | 優先度（high / medium / normal） |
| progress_status | string | ○ | 進捗ステータス（未着 / 仕掛 / 保留 / 完了）（CHG-050） |
| due_date | string\|null | - | 期限日（`YYYY-MM-DD` または `YYYY-MM-DDTHH:MM`）。サブ画面の入力欄手入力と時刻候補選択の両方を受ける |
| list | string\|null | - | 所属リスト名（null = リストなし） |
| parent | string\|null | - | 親タスクID。null:親タスク、値がある場合:サブタスク |
| tags | array | ○ | タグリスト（例：["urgent", "client"]） |
| sort_order | number | ○ | 一覧内の表示順 |
| delete_flag | number | ○ | ゴミ箱フラグ（0:通常, 1:ゴミ箱） |
| created_at | string | ○ | 作成日時（ISO 8601形式） |
| updated_at | string | ○ | 更新日時（ISO 8601形式） |
| completed_at | string\|null | - | 完了日時（ISO 8601形式） |
| deleted_at | string\|null | - | ゴミ箱移動日時（ISO 8601形式） |

### 2.3 本文エリア

フロントマター以下のMarkdownコンテンツを記述。リッチな記述が可能：

```markdown
---
...frontmatter...
---

## 詳細説明

タスクの背景や実装ガイドをMarkdownで記述

- チェックリスト項目1
- チェックリスト項目2

### コード例

```javascript
// 実装例
```

### 2.4 パス解決ルール（CHG-016 / CHG-051）

- タスクファイル正本の frontmatter にはファイル格納パスを保存しない
- `_index.yaml` にも `task_file_path` は保存しない
- `taskService` はファイル走査時に実ファイルパスを `_filePath` などの内部メタデータとして保持する
- 保存・更新・削除・復元・外部アプリ起動は `taskId` から main process が現在の実ファイルパスを解決する
- 旧形式の `task_file_path` が残る既存タスクは、読み込み時に正本から除去する

### 参考
- [リンク](https://example.com)
```

## 3. ステータス定義

| ステータス | 意味 | 用途 |
|-----------|------|------|
| todo | 未着手 | 新規作成時のデフォルト |
| doing | 作業中 | ユーザーが明示的に変更 |
| blocked | ブロック中 | 外部依存により進行不可 |
| done | 完了 | チェックボックスで自動遷移 |

### 状態遷移図

```text
[新規作成] → todo
               │
               ├──→ doing ←──→ blocked
               │      │
               └──────┴──→ done（完了）
                         ↑完了チェック
                         └再度开始で todo に戻す
```

### 遷移ルール

- どのステータスからでも他のステータスへ自由に遷移可能
- チェックボックス操作（✓）：`status → done` / `done → todo`
- ステータス自体は詳細ペイン下部のドロップダウンで変更する

## 4. 機能一覧

### 4.1 タスク作成（クイック追加）

| 項目 | 内容 |
|------|------|
| トリガー | メインペイン上部のクイック追加バーにタスク名を入力しEnter |
| 入力 | タスク名（必須） |
| 初期値 | status: todo, priority: medium, delete_flag: 0, progress_status: 未着 |
| ビュー連動初期値 | 「今日」では due_date=今日、「明日」では due_date=明日、「次の7日間」では due_date=今日 を初期設定し、「すべて」では due_date=null |
| 保存先 | `30_data/tasks/T-XXX.md`（新規ファイル） |
| 処理 | taskService.addTask() で ID 採番、frontmatter 生成、ファイル書き込み |
| 自動実行 | indexService がキャッシュから `_index.yaml` を再生成 |
| UI反映 | watcher が変更を検知 → renderer に通知 → リスト更新 |
| 設計意図 | TickTickのクイック追加方式を参考。最小ステップでタスク登録 |

### 4.2 タスク更新

| 項目 | 内容 |
|------|------|
| トリガー | 詳細ペインで項目を編集（フォーカスアウト） |
| 対象フィールド | title, due_date, priority, list, progress_status, tags |
| 本文編集 | Markdownコンテンツエリアを直接編集。編集モードでも `#`, `##`, `###`, `**強調**`, リスト、リンク、コードなどの Markdown 記法を視覚的に強調し、保存データは従来どおり Markdown テキストのままとする（CHG-060） |
| 日付変更 UI | メインペイン/詳細ペインとも日付表示クリックで年月日時入力サブ画面を開く。上部タブは置かず、月間カレンダー、`今日/明日/1週間後/日付のみ` ショートカット、選択結果の手入力欄、30分単位の時刻候補リストを提供する。`OK` で保存・`クリア` で未設定化・キャンセル時は未保存とする（CHG-039 / CHG-049） |
| 日付表示形式 | 詳細ペインは `MM/DD HH:MM` を基本表示。メインペインは時刻あり `MM/DD HH:MM`、時刻なし `MM/DD` を表示する（CHG-039 / CHG-049） |
| Markdownプレビュー | 詳細ペイン本文は編集モード / プレビューモードを切り替え可能 |
| Markdown編集ハイライト | 編集モードでは VSCode の Markdown 編集画面に近い軽量なシンタックスハイライトを提供する。表示上の装飾のみで、frontmatter や本文保存形式、IPC ペイロードは変更しない（CHG-060） |
| 詳細ペイン右上アクション | 既存プレビュー導線は虫眼鏡アイコン表示とし、その左に詳細ペイン拡大ボタンを配置する。拡大ボタン押下時は詳細ペインをアプリ内の残り全パネル幅へ広げ、再押下で通常幅へ戻す。別途「新しいアプリで開く」アイコンを追加し、対象タスク Markdown を外部アプリで開く。CHG-064 以降は `settings.yaml` の `externalEditorPath` が設定されていればその実行ファイルへ main process が解決した対象 `.md` パスを渡し、未設定時は OS 既定アプリで開く。renderer は外部アプリ起動や実ファイルパス解決を直接行わず、IPC へ `taskId` を渡す。起動失敗時はユーザへエラー通知する（CHG-040 / CHG-059 / CHG-064） |
| 本文リンク起動 | 詳細ペイン本文の Markdown リンクは URL / ローカルファイル / フォルダを判定して起動する。URL は既定ブラウザ、ファイルは既定アプリ、フォルダはエクスプローラで開く。起動失敗時はエラー通知する（CHG-041） |
| タスク行レイアウト | メインペインの進捗ステータスバッジはタスク名の左側に表示する（CHG-039） |
| タスクID表示 | 詳細ペインフッター右端に `ID: T-XXXX` 形式で常時表示する（CHG-031） |
| フッター日時表示 | 詳細ペインフッターに `登録日時：YYYY/MM/DD` を常時表示し、完了タスクのみ `完了日時：YYYY/MM/DD` を追加表示する。完了日時はメタ情報セクションからは削除する（CHG-038） |
| 親タスクリンク | 選択中タスクに `parent` が設定されている場合、詳細ペイン上部に親タスクへ移動できるリンクを表示する。クリック時は該当親タスクを選択し、詳細ペインを親タスク内容へ切り替える（CHG-047） |
| 関連サブタスク表示 | 選択中タスクを親に持つタスクが存在する場合、詳細本文の下に `▼関連サブタスク` セクションを表示する。セクションは折りたたみ可能とし、各行にはチェック、タイトル、進捗、タスクIDを表示する。行クリック時は該当サブタスクを選択する（CHG-047） |
| 保存先 | 対象タスクファイル（frontmatter + 本文を上書き） |
| 処理 | taskService.updateTask() で frontmatter を更新して上書き |
| 自動実行 | indexService が `_index.yaml` を再生成 |
| UI反映 | watcher が通知 → リアルタイム反映 |

### 4.3 タスク完了

| 項目 | 内容 |
|------|------|
| トリガー | チェックボックスクリック（✓） |
| 処理 | status: done, completed_at: 現在日時を frontmatter に書き込み |
| 日付自動設定 | `due_date` が空欄の場合のみ、完了日のローカル日付を `YYYY-MM-DD` で自動設定する（CHG-054） |
| 処理 | taskService が watcher 経由で自動実行 |
| カスケード完了 | 親タスク完了時、5階層以内の未完了子孫タスクも一括で status: done に更新（CHG-009 / CHG-045） |
| 親進捗自動推定 | 親タスクが `未着` の場合のみ、直下サブタスクを起点に5階層以内の子孫状態から `仕掛` または `保留` へ自動更新する。親タスクが `仕掛` / `保留` / `完了` の場合は、子タスク状態で親進捗を上書きしない（CHG-012 / CHG-045 / BUG-20260505-02） |
| 表示 | 一覧から除去（完了一覧に移動）、行が打ち消し線・薄色表示 |
| UI反映 | watcher → リアルタイム反映 |

### 4.4 完了取消

| 項目 | 内容 |
|------|------|
| トリガー | 完了一覧でチェックを外す |
| 処理 | status: todo, completed_at: null を frontmatter に書き込み |
| 処理 | progress_status が「完了」なら「仕掛」に自動変更 |
| カスケード取消 | 親タスクの progress_status を「完了」から戻した場合、5階層以内の子孫タスクの progress_status も同じ方向に戻す（CHG-010 / CHG-045） |
| 親進捗自動推定 | 親タスクが `未着` の場合のみ、子孫タスクに `仕掛` または `完了` が1件でもあれば `仕掛`、完了・仕掛がなく `保留` が1件でもあれば `保留` に再推定する。親タスクが `仕掛` / `保留` / `完了` の場合は、子タスク状態で親進捗を上書きしない（CHG-012 / CHG-045 / CHG-050 / BUG-20260505-02） |

### 4.4.1 親タスク保留時のサブタスク連動

| 項目 | 内容 |
|------|------|
| 親を保留へ変更 | 5階層以内の子孫タスクのうち、未完了かつ `未着` / `仕掛` のタスクを `保留` に更新する |
| 完了子タスク | 親を `保留` にしても、`完了` の子孫タスクは変更しない |
| 保留から未着へ変更 | `保留` の子孫タスクのみ `未着` に戻す。完了子タスクは変更しない |
| 保留から仕掛へ変更 | `保留` の子孫タスクのみ `仕掛` に戻す。完了子タスクは変更しない |
| 保留から完了へ変更 | 既存の親タスク完了カスケードに従い、未完了の子孫タスクを `完了` にする |
| 表示 | 一覧の該当ビューに戻す |
| UI反映 | watcher → リアルタイム反映 |

### 4.8 タスク右クリックメニュー

タスク一覧の各タスク行を右クリックすると、コンテキストメニューを表示する。

| 項目 | 内容 |
|------|------|
| 表示条件 | 通常のタスク一覧でのみ表示。検索ビュー・完了一覧・ゴミ箱では無効 |
| 削除導線 | ごみ箱移動は右クリックメニューからのみ実行し、ホバー時ボタンは表示しない |
| サブタスクの追加 | 対象タスク直下にインライン入力欄表示、Enter でサブタスク作成。5階層目では追加不可（CHG-045） |
| リスト設定 | サブメニューでリスト一覧表示、選択で `list` フィールド変更 |
| タグ設定 | frontmatter `tags: []` を編集（T-032以降） |
| 複製する | タスク全体を複製、タイトル末尾に「（コピー）」を付与 |
| ごみ箱に移動 | delete_flag: 1 に変更 |

### 4.9 サブタスク作成

| 項目 | 内容 |
|------|------|
| データ構造 | サブタスクは独立した `.md` ファイル、frontmatter に `parent: T-XXX` を設定 |
| 入力UI | 対象タスク直下にインライン入力欄を表示。ただし5階層目タスクでは表示しない、または無効化する |
| 表示 | メインペインで対象タスク直下に最大5階層まで再帰的に階層表示 |
| 階層インデント | メインペインの階層表示は、1階層進むごとに同じ幅だけ左インデントを増やす。親→子の初段だけ大きくずらさず、子→孫、孫→ひ孫と同程度の差分に揃える（CHG-048） |
| インデント基準 | 行左端の基本余白は通常タスクと同じにし、階層差分は `--task-depth` に基づく固定ステップで算出する。展開ボタン、ドラッグハンドル、チェックボックス、進捗バッジ、タイトルの横位置が階層ごとに不規則に増えないようにする（CHG-048） |
| 展開状態 | 親タスクごとに展開・折りたたみを切り替え可能 |
| 詳細ペイン表示 | 親があるタスクでは親タスクリンクを表示し、子があるタスクでは `▼関連サブタスク` セクションに直下サブタスクを一覧表示する（CHG-047） |
| 詳細ペイン折りたたみ | `▼関連サブタスク` は開閉可能とし、閉じた場合は `▶関連サブタスク` として見出しのみ表示する（CHG-047） |
| 詳細ペイン遷移 | 親タスクリンクまたは関連サブタスク行をクリックすると、対象タスクを選択し詳細ペインを切り替える。検索・フィルタ中でも対象タスクが存在する限り選択状態を破綻させない（CHG-047） |
| 親進捗反映 | 子孫タスクの `progress_status` 変化に応じて、5階層以内の祖先タスクの `progress_status` / `status` を自動再計算 |
| 補足 | サブタスク追加時は親タスクが自動展開される |

### 4.12 タスクのドラッグ並び替え / セクション移動（CHG-021）

| 項目 | 内容 |
|------|------|
| トリガー | メインペインのタスク行に表示するドラッグハンドルをドラッグ＆ドロップ |
| ドラッグ開始点 | チェックボックス左隣に縦向きのドラッグハンドルを表示し、このハンドルのみをドラッグ起点とする。行全体、チェックボックス、タスク名、期限・進捗バッジなどからはドラッグを開始しない（CHG-046） |
| 対象ビュー | すべて / 今日 / 明日 / 次の7日間 / リスト / リストなし（完了・ゴミ箱・検索は対象外） |
| 並び替え | ドロップ位置に応じて `sort_order` を更新し、同一表示集合で連番再採番 |
| 実行API | 既存の `sort_order` frontmatter と `tasks:reorder` IPC / `taskService.reorderTasks()` を利用し、専用の並び替え用データ構造は追加しない（CHG-046） |
| 日付自動更新 | 今日→明日で `due_date=明日`、明日→今日で `due_date=今日` |
| 進捗自動更新 | CHG-028 により「未着・仕掛」は統合セクションのため、セクション間移動による `progress_status` 自動変更は発生しない |
| 逆遷移 | 日付セクション間の逆方向（明日→今日）は同じルールで適用 |
| 整合制約 | `status=done` タスクは未着/仕掛へのドラッグ変更を禁止 |
| 親子制約 | 親子関係の変更は行わず、初期スコープでは同一親配下の並び替えのみ許可 |
| 失敗時 | 並び順更新と属性更新を一体保存し、部分成功を残さない |

### 4.13 仕掛・保留フィルター（CHG-043 / CHG-050）

| 項目 | 内容 |
|------|------|
| トリガー | 左ナビの「仕掛」フィルター選択 |
| 表示対象 | `progress_status = 仕掛` かつ `delete_flag = 0` のタスク |
| 保留フィルター | 左ナビの「保留」選択時、`progress_status = 保留` かつ `status != done` かつ `delete_flag = 0` のタスクを表示 |
| 配置 | ナビ項目は「仕掛」「保留」「完了」「ゴミ箱」の順に表示 |
| 目的 | 進行中タスクの即時抽出と確認導線の強化 |

### 4.14 破損タスクファイルの検知と警告表示（CHG-044）

AI や外部エディタが `data/tasks/*.md` を直接追加・編集した場合、YAML frontmatter が不正になることがある。
Cotaska は、1 件の破損ファイルでタスク一覧全体を停止させず、破損ファイルを「診断用タスク」として一覧に表示する。

| 項目 | 内容 |
|------|------|
| 検知タイミング | アプリ起動時、ファイル変更検知時、キャッシュ再構築時 |
| 検知対象 | `data/tasks/*.md` の YAML frontmatter |
| 破損条件 | YAML パース失敗、frontmatter 区切り不正、`id` 欠落、`id` 重複 |
| 内部表現 | `is_invalid: true` を持つ診断用タスク |
| 表示 | メインペインに通常タスクと同じ一覧上で表示し、薄い赤背景・赤線・警告マーク・`要確認` バッジで識別する |
| 詳細ペイン | エラー種別、エラー位置、エラー内容、対象ファイルパスを表示する |
| ファイル起動 | 詳細ペイン右上の「新しいアプリで開く」から対象 Markdown を OS 既定アプリで開く |
| 操作制限 | 完了、編集、期限変更、リスト変更、タグ変更、複製、ゴミ箱移動、並び替えは無効 |
| 復旧 | 対象 Markdown を修正し、再読み込みまたは再起動で通常タスクとして読み込めた場合、破損表示は解消される |

破損タスクの主な内部項目:

| フィールド | 説明 |
|-----------|------|
| id | `__INVALID__${相対パス}`。通常の `T-XXXX` と衝突しない診断用 ID |
| title | `読み込みエラー: ファイル名` |
| progress_status | `要確認` |
| tags | `["破損タスク"]` |
| is_invalid | 破損タスク判定フラグ |
| validation_error | YAML パースエラー等の内容 |
| validation_error_name | エラー種別（例: `YAMLException`） |
| validation_error_line | 取得できる場合の行番号 |
| validation_error_column | 取得できる場合の列番号 |
| validation_file_path | 対象 Markdown の絶対パス（診断表示用。frontmatter には保存しない） |

## 5. サービス層インターフェース（taskService.js）

### 初期化

```javascript
await taskService.openTaskService()
// 起動時に 30_data/tasks/ の全ファイルを読み込み、メモリキャッシュを構築
```

### タスク取得

```javascript
tasks = taskService.getAllTasks()
// delete_flag = 0 のタスクを返す

completed = taskService.getCompletedTasks()  
// status = done のタスクを返す（ソート: completed_at DESC）

trashed = taskService.getTrashedTasks()
// delete_flag = 1 のタスクを返す
```

### タスク操作

```javascript
newTask = taskService.addTask({
  title: string,
  due_date?: string,  // YYYY-MM-DD
  priority?: string,  // high|medium|normal
  list?: string,
  parent?: string,    // T-XXX
  tags?: string[]
})
// 新規タスク作成、ID自動採番、ファイル + キャッシュ + _index.yaml を更新

taskService.updateTask(id, {
  title?: string,
  status?: string,
  priority?: string,
  due_date?: string,
  list?: string,
  progress_status?: string,
  sort_order?: number,
  tags?: string[]
  // 本文は含まない（別途編集）
})
// 指定フィールドのみ更新

taskService.reorderTasks({
  orderedIds: string[],
  context?: {
    view: string,
    fromSection?: string,
    toSection?: string
  },
  applyAutoFields?: {
    due_date?: string|null,
    progress_status?: string
  }
})
// 並び順更新と必要な属性自動更新を一括保存

taskService.trashTask(id)      // delete_flag = 1
taskService.restoreTask(id)    // delete_flag = 0
taskService.deleteTask(id)     // ファイルを archive/ に移動

taskService.duplicateTask(id)  // タスク複製
```

### 破損タスク取得時の扱い（CHG-044）

```javascript
tasks = taskService.getAllTasks()
// 正常タスクと、delete_flag=0 の破損タスクを返す
// 破損タスクは is_invalid=true と validation_error を持つ
```

破損タスクは診断用の一時表現であり、通常の更新 API の対象外とする。
`updateTask()`、`completeTask()`、`trashTask()`、`duplicateTask()`、`reorderTasks()` 等では、`is_invalid=true` のタスクを書き戻さない。
対象ファイルの修正は、詳細ペインから外部アプリで Markdown を開いて行う。
外部アプリ起動時は renderer が保存済みパスを渡すのではなく、`taskId` を main process へ渡して現在の実ファイルパスを解決する。

task = taskService.getTaskById(id)
tasks = taskService.getTasksByList(listName)
tasks = taskService.getTasksByParent(parentId)
```

## 7. バリデーション

| 項目 | ルール |
|------|--------|
| title | 1文字以上、200文字以内 |
| priority | high / medium / normal のいずれか |
| status | todo / doing / blocked / done のいずれか |
| due_date | YYYY-MM-DD形式、過去日も許容、null 可 |
| list | 存在するリスト名または null |
| parent | 既存タスクID（T-XXX形式）または null |
| delete_flag | 0 または 1 |
| progress_status | 未着 / 仕掛 / 保留 / 完了 のいずれか |

- 削除済みタスク（delete_flag=1）を親タスクとして指定してはならない
- サブタスクを持つ親タスクをゴミ箱へ移動する場合は、「全て削除」「親だけ削除」「キャンセル」の3択で確認する
  - 「全て削除」では、5階層以内の配下タスクをまとめてゴミ箱へ移動する
  - 「親だけ削除」では、直下の子タスクの `parent` を `null` に更新し、削除済みタスクを親として参照させない
  - 「キャンセル」では、親タスク・配下タスクとも変更しない
- 親子関係は最大5階層まで対応する（CHG-045）。サブタスクのサブタスクも5階層以内であれば有効な親子関係として扱う
- YAML frontmatter が不正なタスクファイルは、通常タスクとして保存・更新せず、`is_invalid=true` の破損タスクとして表示する
- `title` などの YAML 値に `:` を含める場合は引用符で囲む必要がある

## 8. タスク階層上限（CHG-045）

タスクの親子関係は `parent` frontmatter で管理し、最大5階層まで表示・操作対象とする。
サブタスクを親に持つタスクも、5階層以内であれば正式対応対象とする。

| 階層 | 条件 |
|------|------|
| 1階層目 | `parent: null` |
| 2階層目 | 1階層目のタスクを親に持つ |
| 3階層目 | 2階層目のタスクを親に持つ |
| 4階層目 | 3階層目のタスクを親に持つ |
| 5階層目 | 4階層目のタスクを親に持つ |

6階層目以上のタスクは階層上限超過として扱う。

- タスク一覧から除外しない
- 親配下ではなく、1階層目と同じ形で表示する
- 行および詳細ペインに警告を表示する
- タスク自体の編集、完了、削除は原則可能とする
- サブタスク追加によって6階層目以上を新規作成する操作は不可とする

存在しない親ID、循環参照、表示対象外の親を参照するタスクも、一覧から消さずルート相当で表示する。

## 9. 詳細ペイン親子タスク導線（CHG-047）

詳細ペインでは、選択中タスクの `parent` と全タスク一覧から親子関係を解決し、親タスクおよび直下サブタスクへの導線を表示する。

| 項目 | 内容 |
|------|------|
| 親タスクリンク表示条件 | 選択中タスクの `parent` が既存の有効タスクIDを参照している |
| 親タスクリンク表示位置 | 詳細ペインのメタ情報下、タスク詳細本文より上 |
| 親タスクリンク操作 | クリック時に親タスクを選択し、メインペインの選択状態と詳細ペイン内容を更新する |
| 関連サブタスク表示条件 | `parent` が選択中タスクIDと一致する有効タスクが1件以上存在する |
| 関連サブタスク表示位置 | 詳細本文の下 |
| 関連サブタスク見出し | 開状態は `▼関連サブタスク`、閉状態は `▶関連サブタスク` とする |
| 関連サブタスク行 | チェック、タイトル、進捗ステータス、タスクIDを表示する |
| 関連サブタスク操作 | 行クリック時に該当サブタスクを選択し、詳細ペイン内容を更新する。チェック操作は通常の完了/取消操作として扱う |
| 子なしの場合 | セクションを非表示、または追加導線のみの控えめな表示とし、本文編集を邪魔しない |
| 階層上限 | 5階層目タスクではサブタスク追加導線を表示しない、または無効化する |
| 破損タスク | `is_invalid=true` の診断用タスクは親子導線の対象外とする |

本仕様は表示・遷移導線の追加であり、DB/frontmatter の新規フィールドは追加しない。既存の `parent` とタスク一覧取得結果を利用する。

## 10. バージョン情報表示（CHG-057）

Cotaska の現在バージョンは `20_app/package.json` の `version` を正本とし、アプリ内で利用者が確認できるようにする。

### 10.1 表示場所

| 項目 | 内容 |
|------|------|
| 表示場所 | 設定画面の「アプリ情報」セクション |
| 表示形式 | `Cotaska 0.1.0` のように、製品名と現在バージョンを1行で表示する |
| 補足情報 | 必要に応じて配布種別、データ保存場所、リリースノート導線を同じセクションに配置する |
| 設定画面未実装時 | T-0021 の設定画面実装を優先する。先行実装が必要な場合は、main/preload 側の現在バージョン取得 API のみを先に追加してよい |

### 10.2 現在バージョン取得

| 項目 | 内容 |
|------|------|
| 正本 | `20_app/package.json` の `version` |
| 取得方法 | main process が `package.json` の `version` を取得し、preload 経由で renderer へ渡す |
| renderer 制約 | renderer は Node.js API や `package.json` ファイルへ直接アクセスせず、`window.cotaskaAPI` 経由で取得する |
| API 例 | `window.cotaskaAPI.app.getVersion()` |

### 10.3 最新版確認と更新案内

初期対応では自動更新を行わない。Cotaska はローカルタスクデータを正本として扱うため、利用者確認なしの上書き更新よりも、安全な手動更新案内を優先する。

| 項目 | 内容 |
|------|------|
| 最新版取得元 | GitHub Releases または `latest.json` |
| 比較対象 | 現在バージョンと最新バージョン |
| 最新版あり | 「最新版があります。ダウンロードページを開きますか？」という確認導線を表示する |
| 最新版なし | 「最新です」などの控えめな状態表示に留める |
| 取得失敗 | ネットワークエラー等を通知しつつ、通常利用は妨げない |
| 自動更新 | 初期スコープ外 |

## CHG-061 タスク詳細文字サイズ変更

タスク詳細ペインの本文表示、本文編集モード、Markdownプレビューは、ユーザー操作により文字サイズを変更できる。

| 項目 | 内容 |
|------|------|
| 対象範囲 | タスク詳細ペインの本文表示、Markdown編集ハイライト、Markdownプレビュー |
| 操作方法 | `Ctrl` + `+` で拡大、`Ctrl` + `-` で縮小、`Ctrl` + マウスホイールで拡大・縮小 |
| UI補助 | 詳細ペイン右上に `A-` / `A+` と現在の px 値を表示する |
| サイズ範囲 | 12px から 22px まで。範囲外には変化しない |
| 設定保持 | renderer の `localStorage` に `cotaska.detailContentFontSize` として保持する。タスクMarkdown、frontmatter、DB、IPCには保存しない |
| 既存機能との関係 | CHG-059 の詳細ペイン拡大表示中も同じ設定を適用する。CHG-060 のMarkdown編集ハイライトは textarea と背景ハイライトへ同じ文字サイズを適用し、スクロール同期を維持する |

### CHG-061 追補：文字サイズ変更時の現在値表示

`Ctrl` + マウスホイール、`Ctrl` + `+`、`Ctrl` + `-` で文字サイズを変更した直後、詳細ペイン上に `文字サイズ NNpx` を一時表示する。

| 項目 | 内容 |
|---|---|
| 表示位置 | 詳細ペイン右上寄り。ヘッダー直下に重ねて表示し、本文編集領域の入力を妨げない |
| 表示内容 | `文字サイズ NNpx` |
| 表示時間 | 最後の変更操作から約1秒後に自動で消える |
| 保存 | 表示そのものは保存しない。文字サイズ値の保存方針は CHG-066 に従う |

### CHG-066 文字サイズ設定保存への移行

CHG-066 以降、タスク詳細ペインの文字サイズは設定画面の「文字サイズ」と同じ値として扱い、正式保存先を `settings.yaml` の `detailTextSize` に移行する。

| 項目 | 内容 |
|------|------|
| 保存先 | `settings.yaml` の `detailTextSize` |
| 反映対象 | タスク詳細本文、本文編集、Markdownプレビュー、Markdown編集ハイライト |
| サイズ範囲 | 10px から 28px まで |
| 既存ショートカット | `Ctrl` + `+` / `Ctrl` + `-` / `Ctrl` + マウスホイールは同じ設定値を更新し、保存成功後に即時反映する |
| 互換 | 既存 localStorage 値は初回移行元として扱い、タスクMarkdown、frontmatter、`_index.yaml` には保存しない |
