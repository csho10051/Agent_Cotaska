## 現象

設定 > アプリ情報で「再起動して更新」をクリックすると Cotaska は終了するが、更新後に再起動しない。
再度起動してもアプリのバージョンが上がっていない。

## 再現手順

1. 旧バージョンの Cotaska-Portable を起動する。
2. 設定 > アプリ情報を開く。
3. 「更新を確認」を実行する。
4. 更新ファイルをダウンロードする。
5. 「再起動して更新」をクリックする。
6. アプリ終了後の再起動有無と、再起動後のバージョン表示を確認する。

## 期待動作

「再起動して更新」をクリックすると、Cotaska 終了後に Portable 更新スクリプトが実行され、`Cotaska-Portable.zip` の内容でアプリ本体が差し替えられる。
更新完了後は `Cotaska.exe` が再起動し、アプリ情報のバージョンが最新になる。

## 調査メモ

- GitHub Releases latest は `v0.2.2` で、assets に `Cotaska-Portable.zip` と `Cotaska-Portable.zip.sha256` が存在することを確認した。
- ユーザー環境のアプリログでは、対象 Portable 環境が `0.2.1` のまま起動している記録が残っていた。
- `%TEMP%\Cotaska-updates\0.2.2\` に `Cotaska-Portable.zip` と `apply-portable-update.ps1` が残っており、ダウンロードと更新スクリプト生成までは到達していた。
- 一方で、対象 Portable 環境の `logs/` に `portable-update-*.log` は作成されていなかった。更新スクリプトが正常に開始できていない、またはログ出力前後で異常終了している可能性が高い。
- 生成された `apply-portable-update.ps1` は `fs.writeFileSync(scriptPath, ..., "utf8")` で UTF-8 BOMなしとして出力され、`powershell.exe` で実行されている。
- Windows PowerShell 5.1 は UTF-8 BOMなしの `.ps1` を ANSI として読むため、日本語を含む `$PortableRoot` が `Cotaska_タスク管理ツール` から `Cotaska_繧ｿ繧ｹ繧ｯ邂｡逅・ヤ繝ｼ繝ｫ` に文字化けすることを再現確認した。
- そのため、日本語パス配下の Portable 環境では更新先パスが壊れ、差し替え・再起動に失敗する可能性が高い。
- さらに `spawn("powershell.exe", ...)` の `error` / `exit` を記録しておらず、`stdio: "ignore"` のため、更新スクリプト自体の失敗理由がアプリログに残らない。
- 実行中の Electron は通常複数の `CotaskaCore.exe` プロセスを持つ。更新スクリプトは `Get-Process -Name "Cotaska","CotaskaCore"` が0件になるまで待つため、終了後に補助プロセスが残る場合はタイムアウト要因にもなり得る。

## 推定原因

主因候補は、Portable 更新スクリプトの文字コードと実行方式の不整合。

`20_app/src/main/main.js` の `createPortableUpdaterScript()` は日本語パスを含む PowerShell スクリプトを UTF-8 BOMなしで保存しているが、実行には Windows PowerShell 5.1 の `powershell.exe` を使っている。Windows PowerShell 5.1 は UTF-8 BOMなしを既定ANSIとして解釈するため、`$PortableRoot` が文字化けし、更新対象フォルダを正しく参照できない。

加えて、外部 updater の開始・終了・失敗をアプリ側がログに残していないため、失敗時に利用者へ原因が伝わらない。

## 修正方針

1. 更新スクリプトを Windows PowerShell 5.1 でも安全に読める UTF-8 BOM付き、または UTF-16LE で保存する。
2. 可能であれば `powershell.exe -EncodedCommand` または `pwsh` 検出など、非ASCIIパスに強い起動方式へ変更する。
3. updater 起動時に temp 側にもログを出し、`spawn` の `error` / `exit` をアプリログへ記録する。
4. 更新適用前に `$PortableRoot` の存在、`Cotaska.exe` / `_app` の存在を検証し、失敗時はユーザーが確認できるログを残す。
5. 待機対象プロセスを必要最小限にし、タイムアウト時にもログを残す。
6. 日本語パス配下の Portable 環境で、0.2.1 相当から 0.2.2 以上へ更新できることを確認する。

## 対応内容

- `apply-portable-update.ps1` を UTF-8 BOM付きで生成するよう変更した。
- updater ログを Portable 側 `logs/portable-update-*.log` と、temp 側 `%TEMP%\Cotaska-updates\<version>\portable-update-*.log` の両方へ出力するよう変更した。
- updater 起動時に `scriptPath`, `portableRoot`, `version`, `zipPath` をアプリログへ記録するよう変更した。
- updater process の `error` / `exit` をアプリログへ記録するよう変更した。
- `$PortableRoot` と `$ZipPath` の存在確認を更新開始直後に追加した。
- 待機対象プロセスを、同じ PortableRoot 配下の `Cotaska.exe` / `CotaskaCore.exe` に限定した。
- プロセス待機中とタイムアウト時に対象プロセスIDを updater ログへ出力するよう変更した。

## 2026-05-19 再調査

### 追加で確認した現象

- 0.2.3 更新操作後、`%TEMP%\Cotaska-updates\0.2.3\Cotaska-Portable.zip` と `apply-portable-update.ps1` は生成されていた。
- アプリログでは `Portable updater process started` の直後に `Portable updater process exited before app quit` が `code:0` で記録されていた。
- しかし、temp 側にも Portable 側にも `portable-update-*.log` が作成されていなかった。
- 同じ `apply-portable-update.ps1` を手動で `powershell.exe -File` 実行すると、ログを出力し、実行中 Cotaska プロセスの待機に入った。

### 追加原因

Electron/Node から `spawn("powershell.exe", ["-File", scriptPath], { detached: true, stdio: "ignore" })` で更新スクリプトを直接起動すると、PowerShell が即時 `code:0` で終了し、スクリプト本体が実行されないことを最小再現した。

また、生成された PowerShell スクリプトでは、PortableRoot prefix の末尾に `\` を付ける意図の箇所が JS template literal のエスケープにより `EndsWith("")` / `+ ""` になっていた。これは今回の即終了の主因ではないが、同一prefixの別パスを誤検出し得るため修正対象とした。

### 追加修正

- updater 起動方式を、Electron から detached PowerShell を直接起動する方式から、短命の PowerShell wrapper が `Start-Process` で実 updater を起動する方式へ変更した。
- wrapper は Windows PowerShell の絶対パスを使い、`-EncodedCommand` で起動することで、日本語パスと PATH 解決への依存を減らした。
- 実 updater は従来どおり `apply-portable-update.ps1` を `-File` で実行し、Cotaska プロセス終了を待って更新する。
- PortableRoot prefix の末尾区切りは `[System.IO.Path]::DirectorySeparatorChar` を使うようにした。

### 追加検証

- Node.js から修正後と同等の wrapper 起動を実行し、wrapper は `code:0` で終了、別 PowerShell プロセスとして実 updater が残ることを確認した。
- 実 updater が `%TEMP%\Cotaska-updates\0.2.3\portable-update-*.log` を作成し、実行中 Cotaska プロセスの待機に入ることを確認した。
- `node --check 20_app/src/main/main.js` が成功した。
- Node.js v24.14.0 を PATH 先頭に設定し、`npm run build` が成功した。

### 残確認

- 実際のアプリ更新操作で、0.2.2 から 0.2.3 以上へ更新され、再起動後のアプリ情報が新バージョンになること。

## 2026-05-19 CotaskaUpdater.exe C# fallback 追加方針

PowerShell updater は今回の修正で改善したが、実行ポリシー、PowerShell 5.1 / 7 差分、文字コード、`detached` 起動挙動などの環境要因を受けやすい。
利用者向け仕様は変えず、更新処理の実装方式を堅牢化する内部修正として C# fallback 製 `CotaskaUpdater.exe` を追加する。

### 方針

- C# fallback 製 updater を `tools/CotaskaUpdater.exe` として Portable 配布物へ同梱する。
- Cotaska 本体は `CotaskaUpdater.exe` が存在する場合、これを temp 更新作業フォルダへコピーしてから起動する。
- temp コピーを起動する理由は、updater 実行中に配布物側の `tools/` を安全に差し替えるため。
- `CotaskaUpdater.exe` が存在しない場合は、当面 PowerShell updater にフォールバックする。
- C# updater は zip 展開、バックアップ、`Cotaska.exe` / `_app` / `tools` の差し替え、再起動、temp/Portable 両方へのログ出力を担当する。

### 実装内容

- `20_app/setup/updater/UpdaterFallback.cs` を追加した。
- `20_app/setup/updater/build.ps1` を追加し、`csc.exe` でビルド成果物を `20_app/scripts/CotaskaUpdater.exe` へ出力するようにした。
- `release-all.ps1` に updater ビルドステップを追加した。
- `release-all.ps1` の tools 同期対象に `CotaskaUpdater.exe` を追加した。
- `20_app/src/main/main.js` は `tools/CotaskaUpdater.exe` が存在する場合は updater exe を優先し、なければ PowerShell updater を使うよう変更した。

### 検証状況

- `20_app/setup/updater/build.ps1` が成功し、`20_app/scripts/CotaskaUpdater.exe` が生成されることを確認した。
- `release-all.ps1 -Version "0.2.3"` が成功し、`20_app/release/Cotaska-Portable/tools/CotaskaUpdater.exe` が同梱されることを確認した。
- `node --check 20_app/src/main/main.js` が成功した。
- Node.js v24.14.0 を PATH 先頭に設定し、`npm run build` が成功した。
