# CHG-029: 配布フォルダ構成整理と設定外部化

## 目的

リリース配布物（`Cotaska-0.1.0-dist`）のフォルダ構成を整理し、ユーザーが直接編集可能なデータ領域を明確に分離する。併せて、アプリログと設定データを外部化し、アップグレード時のデータ永続化と初期化処理を簡潔にする。

---

## 現状の問題点

### 現在の構成
```
Cotaska-0.1.0-dist/
├── _app/
│   ├── locales/
│   ├── resources/
│   │   ├── 30_data/          ← タスクデータが asar 内に埋め込まれている
│   │   └── app.asar
│   ├── workspace/logs/       ← ログが workspace 配下に分散
│   └── [DLL群]
├── Cotaska.exe
└── launcher.log
```

### 課題
1. **ユーザがタスクデータを編集できない**
   - `30_data/` が asar 内に埋め込まれており、展開・再パック手順が複雑
   - ユーザが手軽に YAML を編集できない

2. **ログと実行環境が混在**
   - `workspace/logs/` の配置が不明確
   - トラブルシューティング時にユーザがログを見つけにくい

3. **アップグレード時の手順が複雑**
   - リリース配布物全体を差し替える場合、ユーザーデータの保全が曖昧
   - 個別の `_app/` 更新が難しい

---

## 変更内容

### 新フォルダ構成
```
Cotaska-0.1.0-dist/
├── _app/                     (実行バイナリのみ)
│   ├── locales/              (60言語 pak ファイル)
│   ├── app.asar              (アプリ本体バンドル)
│   └── [DLL群]               (chrome, d3d, ffmpeg, libGL など)
├── data/                      ★ ユーザデータ領域
│   ├── tasks/
│   │   ├── T-0001.md
│   │   ├── T-0002.md
│   │   └── _index.yaml
│   └── lists.yaml
├── logs/                      ★ アプリログ
│   └── (app-YYYY-MM-DD.log など)
├── Cotaska.exe                (ランチャー実行ファイル)
├── launcher.log               (ランチャーログ)
└── README.txt                 ★ ユーザ向けガイド
```

### パス変更の詳細

| 項目 | 変更前 | 変更後 | 用途 |
|------|--------|--------|------|
| タスクデータ | `_app/resources/30_data/` | `data/` | ユーザー直接編集対象 |
| アプリログ | `_app/workspace/logs/` | `logs/` | トラブルシューティング |
| app.asar | `_app/resources/app.asar` | `_app/app.asar` | 実行バイナリ |

## 技術仕様

### 1. ソースコード修正

#### 1.1 `main.js` (アプリログパス)
**変更前:**
```javascript
const LOG_DIR = path.join(__dirname, '../workspace/logs');
```

**変更後:**
```javascript
const LOG_DIR = path.join(app.getPath('appData'), '../Cotaska/logs');
// または（相対参照の場合）
const LOG_DIR = path.join(process.cwd(), '../logs');
```

**補記:** Electron の `app.getAppPath()` は実行ファイルのディレクトリを返すため、相対パス `../logs` で配布ルート直下の `logs` フォルダを参照可能。

#### 1.2 `taskService.js` (タスクデータパス)
**変更前:**
```javascript
const TASKS_DIR = path.join(__dirname, '../../..', '30_data/tasks');
```

**変更後:**
```javascript
const TASKS_DIR = path.join(process.cwd(), '../data/tasks');
// または
const TASKS_DIR = path.join(app.getAppPath(), '../data/tasks');
```

#### 1.3 `taskService.js` (_index.yaml パス)
**変更前:**
```javascript
const INDEX_PATH = path.join(__dirname, '../../..', '30_data/tasks/_index.yaml');
```

**変更後:**
```javascript
const INDEX_PATH = path.join(TASKS_DIR, '_index.yaml');
```

#### 1.4 `taskService.js` (lists.yaml パス)
**変更前:**
```javascript
const LISTS_PATH = path.join(__dirname, '../../..', '30_data/lists.yaml');
```

**変更後:**
```javascript
const LISTS_PATH = path.join(process.cwd(), '../data/lists.yaml');
// または
const LISTS_PATH = path.join(app.getAppPath(), '../data/lists.yaml');
```

### 2. 初期化ロジック

`main.js` の `app.ready()` イベントハンドラに以下を追加：

```javascript
async function ensureDataDirectories() {
  const dataDir = path.join(process.cwd(), '../data');
  const tasksDir = path.join(dataDir, 'tasks');
  const logsDir = path.join(process.cwd(), '../logs');

  // ディレクトリ作成
  if (!fs.existsSync(dataDir)) fs.mkdirSync(dataDir, { recursive: true });
  if (!fs.existsSync(tasksDir)) fs.mkdirSync(tasksDir, { recursive: true });
  if (!fs.existsSync(logsDir)) fs.mkdirSync(logsDir, { recursive: true });

  // デフォルト lists.yaml が無い場合は作成
  const listsPath = path.join(dataDir, 'lists.yaml');
  if (!fs.existsSync(listsPath)) {
    const defaultLists = { lists: [], tags: [] };
    fs.writeFileSync(listsPath, YAML.dump(defaultLists), 'utf-8');
  }

  // デフォルト _index.yaml が無い場合は作成
  const indexPath = path.join(tasksDir, '_index.yaml');
  if (!fs.existsSync(indexPath)) {
    const defaultIndex = { tasks: [], task_file_roots: ['.'], next_task_id: 1, last_updated: new Date().toISOString() };
    fs.writeFileSync(indexPath, YAML.dump(defaultIndex), 'utf-8');
  }
}

// app.ready() 内で呼び出し
app.on('ready', async () => {
  await ensureDataDirectories();
  // ... 既存の処理
});
```

### 3. ビルド設定変更

#### 3.1 `package.json` (electron-builder)
**変更前:**
```json
"build": {
  "files": [
    "dist/**/*",
    "resources/**/*"
  ],
  "extraMetadata": {
    "main": "dist/main/main.js"
  }
}
```

**変更後:**
```json
"build": {
  "files": [
    "dist/**/*"
  ],
  "extraMetadata": {
    "main": "dist/main/main.js"
  }
}
```

**補記:** `resources/30_data/` を除外し、ビルド時に `_app/` には app.asar のみが含まれるようにする。

#### 3.2 リリース配布パッケージング
ビルド後、以下の構成を手動で作成（またはスクリプト化）：

```powershell
# 配布フォルダ構成スクリプト例
$distDir = "release/Cotaska-0.1.0-dist"
$appDir = Join-Path $distDir "_app"

# 既存ファイル（_app 内容）は維持
# 新規に以下を追加
New-Item -ItemType Directory -Path "$distDir/data/tasks" -Force > $null
New-Item -ItemType Directory -Path "$distDir/logs" -Force > $null

# デフォルト lists.yaml 作成
"`nlist: []`ntags: []" | Set-Content "$distDir/data/lists.yaml"

# README.txt 作成
@"
【Cotaska の使い方】

1. Cotaska.exe をダブルクリックして起動してください。

2. data/ フォルダ内のタスクファイルが自動で読み込まれます。
   - tasks/T-XXXX.md: タスクファイル
   - lists.yaml: リスト定義ファイル

3. トラブルが発生した場合は logs/ フォルダの記録を確認してください。

4. app.asar と DLL ファイルは更新用です。
   手動での編集は不要です。
"@ | Set-Content "$distDir/README.txt"
```

追補（2026-03-25）:
- タスク追加時に `_app/resources/30_data/` 側へ Markdown が生成される漏れを確認。
- 原因は `indexService.js` の参照先が旧 `30_data/tasks` 固定だったこと。
