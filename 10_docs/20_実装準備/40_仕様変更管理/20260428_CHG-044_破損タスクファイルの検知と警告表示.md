# 仕様変更ファイル

## 変更ID

CHG-044

## 変更内容

`data/tasks/*.md` のタスクファイル読み込み時に YAML frontmatter の破損を検知し、アプリ全体のタスク読み込みを停止せず、破損タスクとしてタスク一覧に表示する。

- 起動時・ファイル変更検知時にタスク Markdown を検証する
- YAML パースに失敗したファイルは通常タスクとして読み込まず、破損タスクとして仮登録する
- 破損タスクはタスク一覧に表示し、赤系の背景・警告アイコン・「要確認」相当の状態で識別できるようにする
- 詳細ペインにエラー内容、対象ファイルパス、可能なら行番号・列番号を表示する
- 詳細ペインから対象ファイルを開ける導線を用意する
- 破損タスクが存在しても、他の正常なタスクは通常通り表示する

想定する破損タスクの内部表現例:

```js
{
  id: "__INVALID__T-0002.md",
  title: "読み込みエラー: T-0002.md",
  status: "todo",
  priority: "high",
  progress_status: "要確認",
  delete_flag: 0,
  is_invalid: true,
  validation_error: "YAMLException: ... line 3, column 18",
  task_file_path: "D:/.../data/tasks/T-0002.md"
}
```

### 破損タスク検知仕様

破損タスクは、Cotaska が `data/tasks/*.md` を読み込む際に、通常タスクとして安全に扱えないファイルを画面上で可視化するための診断用タスクである。

#### 破損タスクとして扱う条件

- Markdown ファイルの YAML frontmatter を `gray-matter` / `js-yaml` でパースできない
- frontmatter の区切り `---` が不正で、タスクメタ情報を読み取れない
- YAML としては読めるが、`id` が存在しない、または空である
- YAML としては読めるが、`id` が既存タスクと重複しており、どちらかを通常タスクとして一意に扱えない
- `task_file_path` が不正でも、実ファイルパスから復旧可能な場合は破損扱いではなく自動補完対象とする

#### 破損タスクの内部項目

破損タスクは通常タスクと同じ一覧に載せるため、最低限のタスク項目を持つ。ただし、保存・完了・削除などの通常更新対象にはしない。

| 項目 | 内容 |
|------|------|
| `id` | `__INVALID__${相対パス}`。通常の `T-XXXX` と衝突しない値 |
| `title` | `読み込みエラー: ファイル名` |
| `status` | `todo` |
| `priority` | `high` |
| `progress_status` | `要確認` |
| `delete_flag` | `0` |
| `list` | `null` または `Cotaska`。実装時に UI 上の収まりで決定 |
| `tags` | `["破損タスク"]` |
| `is_invalid` | `true` |
| `validation_error` | 例外メッセージ。行番号・列番号が取れる場合は含める |
| `validation_error_name` | 例: `YAMLException` |
| `validation_error_line` | 取得できる場合のみ行番号 |
| `validation_error_column` | 取得できる場合のみ列番号 |
| `task_file_path` | 実ファイルの絶対パス |
| `_filePath` | main process 内部用の実ファイルパス |
| `content` | 空文字、またはエラー説明文 |

#### 表示・操作可否

- タスク一覧には通常タスクと同じ並びに表示する
- 行の背景を薄い赤、左端を赤線、タイトル先頭に警告アイコンを表示する
- 進捗バッジは `要確認` として表示する
- 詳細ペインにはエラー内容、対象ファイルパス、行番号・列番号を表示する
- 詳細ペインから対象 Markdown ファイルを既定アプリで開ける
- 破損タスクでは以下の通常操作を無効化する
  - 完了チェック
  - 進捗ステータス変更
  - タイトル・本文・期限・リスト・タグの保存
  - サブタスク追加
  - 並び替え
  - 複製
  - ゴミ箱移動
- 破損ファイルを削除したい場合は、詳細ペインからファイルを開く、またはエクスプローラーで直接対応する

#### 復旧条件

- 対象ファイルを修正し、次回のファイル監視イベントまたは Cotaska 再起動で YAML パースに成功した場合、破損タスク表示は消える
- 修正後の frontmatter に通常の `id: T-XXXX` が存在する場合、その ID の通常タスクとして表示する
- 修正後も `id` が空・重複している場合は、引き続き破損タスクとして表示する
- 破損タスクは診断用の一時表現であり、破損タスク自身を Markdown として新規保存しない

---

## 変更理由

- AI や外部エディタがタスク Markdown を直接追加・編集した場合、YAML frontmatter が不正になることがある
- 現状は 1 ファイルの YAML パース失敗でタスクサービス初期化が止まり、タスク一覧全体が表示されなくなる
- 壊れたファイルが画面に出ないため、利用者が原因ファイルに気づきにくい
- 破損ファイルを一覧上で可視化することで、原因特定と復旧を容易にする

---

## 影響範囲

- タスク読み込み: `taskService.js` の Markdown/YAML 読み込み処理
- ファイル監視: `watcher.js` 経由の再読み込み・通知処理
- UI: メインペインのタスク行表示、詳細ペインのエラー表示
- IPC/API: `tasks:getAll` の戻り値に `is_invalid` 等の検証情報を含む可能性あり
- データ形式: 既存タスクファイルの形式は変更しない
- 既存機能影響: 正常タスクの表示・編集・完了・削除操作に回帰がないこと
