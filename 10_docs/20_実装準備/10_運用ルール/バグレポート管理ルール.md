# バグレポート管理ルール

## 目的

バグ発見から修正完了までのトレーサビリティを確保し、人間とAIエージェントが同じ情報に基づいて対応できるようにする。

---

## 1. 格納先

```
Cotaska\10_docs\30_実装・検証\10_不具合対応\
```

---

## 2. ファイル命名規則

```
YYYYMMDD_連番_タイトル.md
```

例：`20260317_01_クイック追加でタスクが登録されない.md`

### ルール

- `YYYYMMDD` — 起票日を使用する
- `連番` — 同一日付内で `01` から昇順で採番する（2桁固定）
- `タイトル` — 現象を端的に表す日本語とする
- 拡張子は `.md` とする

---

## 3. バグレポートの記載内容

バグレポートファイルには以下の項目を記載する。

```markdown
# バグレポート

## 基本情報

| 項目 | 内容 |
|------|------|
| 番号 | BUG-YYYYMMDD-XX |
| 報告日 | YYYY-MM-DD |
| 報告者 | （名前またはエージェント名） |
| 対象機能 | （機能名） |
| 重要度 | 高 / 中 / 低 |
| 状態 | 調査中 / 対応中 / 解決済み |

## 現象

（何が起きているか）

## 再現手順

1. 
2. 
3. 

## 期待動作

（本来どう動くべきか）

## 調査メモ

（原因・調査結果）

## 修正方針

（対応方法）

## 関連ファイル

- `ファイルパス` — 説明
```

---

## 4. タスク正本の作成

バグレポート起票後、`00_mgmt/Cotaska_タスク管理ツール/data/tasks/T-XXXX.md` にバグ対応タスクを作成する（**正本**）。タスクの状態管理は task ファイルで完結する。

---

## 5. ステータス管理

| ラベル | 意味 | 次のアクション |
|--------|------|----------------|
| `[調査中]` | 原因調査中 | 原因特定後 → `[対応中]` に変更 |
| `[対応中]` | 修正実装中 | 修正・確認完了後 → `[解決済]` に変更 |
| `[解決済]` | 修正完了・確認済み | バグレポートファイルの「状態」も更新する |

### 運用ルール

1. バグ報告を受領し、まず `Cotaska\10_docs\30_実装・検証\10_不具合対応\YYYYMMDD_XX_タイトル.md` にバグレポートファイルを作成する。
2. バグレポートの「状態」は `調査中` で開始し、現象・再現手順・期待動作を記録する。
3. 実装タスクを `00_mgmt/Cotaska_タスク管理ツール/data/tasks/T-XXXX.md` として個別に作成する（**正本**）。タスクには以下を含める。
   - 設計書反映タスク（設計書・運用ルール・仕様記述の更新が必要な場合）
   - 調査・実装タスク（BUG-YYYYMMDD-XX をタイトルまたはタグに記載）
   - テスト・ユーザ確認タスク
4. 原因特定後、バグレポートの「状態」を `対応中` に更新し、調査メモ・修正方針・関連ファイルを追記する。
5. 実装・テスト後、対応タスクの frontmatter を更新する。
   - `status: done`
   - `progress_status: 完了`
   - `completed_at`
   - `updated_at`
6. 解決時は、バグレポートファイル内の「状態」セルを `解決済み` に更新し、検証結果を追記する。
7. 重要な技術判断や仕様判断がある場合は、`00_mgmt/DECISIONS.md` に意思決定要点を1-3行で記録する。
