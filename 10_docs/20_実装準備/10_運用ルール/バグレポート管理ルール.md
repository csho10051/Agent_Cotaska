# バグレポート管理ルール

## 目的

バグ発見から修正完了までのトレーサビリティを確保し、人間とAIエージェントが同じ情報に基づいて対応できるようにする。

---

## 1. 格納先

```
CoTasker\10_docs\30_実装・検証\10_不具合対応\
```

---

## 2. ファイル命名規則

```
YYYYMMDD_連番_タイトル.md
```

例：`20260317_01_クイック追加でタスクが登録されない.md`

### ルール

- `YYYYMMDD` — 起票日を使用する
- `連番` — 同一日付内で `01` から昇順で採番する（2桁固定）
- `タイトル` — 現象を端的に表す日本語とする
- 拡張子は `.md` とする

---

## 3. バグレポートの記載内容

バグレポートファイルには以下の項目を記載する。

```markdown
# バグレポート

## 基本情報

| 項目 | 内容 |
|------|------|
| 番号 | BUG-YYYYMMDD-XX |
| 報告日 | YYYY-MM-DD |
| 報告者 | （名前またはエージェント名） |
| 対象機能 | （機能名） |
| 重要度 | 高 / 中 / 低 |
| 状態 | 調査中 / 対応中 / 解決済み |

## 現象

（何が起きているか）

## 再現手順

1. 
2. 
3. 

## 期待動作

（本来どう動くべきか）

## 調査メモ

（原因・調査結果）

## 修正方針

（対応方法）

## 関連ファイル

- `ファイルパス` — 説明
```

---

## 4. CURRENT_SPRINT.md への登録

バグレポート起票後、必ず `CURRENT_SPRINT.md` の **「不具合対応タスク」セクション** に追加すること。

### 記載フォーマット

```markdown
### BUG-YYYYMMDD-XX　[状態] タイトル

* 報告日: YYYY-MM-DD
* 状態: 調査中 / 対応中 / 解決済み
* 対象ファイル: `ファイルパス`
* バグレポート: `10_docs/30_実装・検証/10_不具合対応/YYYYMMDD_XX_タイトル.md`
* 概要: 現象の一行説明
```

---

## 5. ステータス管理

| ラベル | 意味 | 次のアクション |
|--------|------|----------------|
| `[調査中]` | 原因調査中 | 原因特定後 → `[対応中]` に変更 |
| `[対応中]` | 修正実装中 | 修正・確認完了後 → `[解決済]` に変更 |
| `[解決済]` | 修正完了・確認済み | バグレポートファイルの「状態」も更新する |

### 運用ルール

- バグ起票時：必ず `[調査中]` から開始する
- 解決時：`CURRENT_SPRINT.md` のステータス更新と同時に、バグレポートファイル内の「状態」セルも `解決済み` に更新する
- 解決済みバグは `CURRENT_SPRINT.md` から削除せず、状態を `[解決済]` のまま残す
